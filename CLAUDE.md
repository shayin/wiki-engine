# AI Wiki 治理文件

## 核心理念

用户只做三件事：**存、问、研究**。中间的整理、调研、分析，全是 AI 的事。

## 目录结构

```
├── inbox/              # 待处理（文章入口）
├── raw/                # 原文归档（按月 YYYY-MM/，不可变）
├── context/            # 个人上下文（用户维护，AI 只读）
│   ├── finance.md      #   投资相关：持仓、风险偏好、关注领域
│   └── ...             #   可按需扩展其他上下文文件
├── decisions/          # 决策日志（AI 写入，用户回看）
│   └── YYYY-MM-DD-简短标识.md
├── todos/              # AI 待办管理
│   ├── active.md       #   待办 + 跟踪项（按课题分组）
│   └── archive/        #   按周归档（YYYY-WXX.md）
├── .cron/              # 定时任务（配置 + 日志 + 脚本）
│   ├── config.sh       #   用户配置（Bark key、调度时间）
│   ├── pending.md      #   通知队列（定时任务写入）
│   ├── logs/           #   执行日志
│   └── scripts/        #   定时任务脚本
└── wiki/               # AI 维护的知识库
    ├── index.md        # 总索引（AI 入口点，每次操作先读此文件）
    ├── log.md          # 处理日志（只追加）
    ├── sources/        # 文章摘要卡片（一对一，每篇存入的文章一张）
    ├── topics/         # 专题内聚合（同标签文章跨篇总结）
    ├── connections/    # 跨课题关联（不同专题间的交叉分析，产出新洞察）
    │   ├── README.md   #   目录说明和格式定义
    │   ├── index.md    #   关联索引（按类型分组）
    │   └── *.md        #   关联分析卡片
    ├── learning/       # 学习计划（把知识库内容组织成有序学习路径）
    │   ├── README.md   #   目录说明
    │   └── *.md        #   学习计划（四阶段：基础→深入→关联→内化）
    └── analysis/       # 课题研究工作区
        └── {课题名}/
            ├── plan.md       # 研究计划（问题定义、已知/未知、搜索方向）
            ├── materials/    # 搜集的原始资料（网页抓取等）
            ├── notes.md      # 研究笔记（中间产出、信息缺口记录）
            ├── report.md     # 最终研究报告（结论、来源、未解决问题）
            └── follow-ups/   # 持续跟踪项（研究产出中需要跟进的部分）
```

## 文章处理流程

1. 新文章进入 `inbox/`（来源：微信转发、浏览器插件、手动放入）
2. AI 执行 `/digest` 处理：
   - 读取文章全文
   - 生成知识卡片到 `wiki/sources/`
   - 原文移到 `raw/YYYY-MM/`
   - 更新 `wiki/index.md` 索引
   - 追加日志到 `wiki/log.md`
   - 检查是否需要更新已有 topic 或创建新 topic

## 知识卡片格式（sources/）

```markdown
---
title: 文章标题
date: YYYY-MM-DD
tags: [标签1, 标签2]
source: article | self
origin: 原文URL（如有）
related: [[关联页面1]], [[关联页面2]]
---

## 一句话
50字以内的摘要。

## 关键要点
- 要点1
- 要点2
- 要点3

## 原文
→ [[raw/文件名]]
```

`source: self` 的文章（用户自己写的）：保留原文，不重写摘要，只做分类打标签和建立关联。

## 主题页面格式（topics/）

当同类标签的文章之间有**实质性关联**时（不只是标签重叠，而是内容有交叉、互补或对比价值），自动生成或更新主题页面。两篇文章有关联即可创建，无需等待固定数量。标签重叠但内容无实质关联时不创建。

```markdown
---
title: 主题名称
tags: [标签]
updated: YYYY-MM-DD
related: [[相关文章1]], [[相关文章2]]
---

## 概述
基于 N 篇文章的聚合总结。

## 核心发现
- ...

## 关联文章
- [[文章1]] - 一句话摘要
- [[文章2]] - 一句话摘要
```

## 课题研究流程

用户说"研究 XX"时触发。Research 产出分两类：

- **知识型**：产出通用方法论、原理等长期有价值的知识 → 入库到 sources/
- **决策型**：产出时效性分析、个人决策建议等 → 保留在 analysis/，不入库

### 流程

1. **开题**：在 `wiki/analysis/{课题名}/` 下创建 `plan.md`
   - 定义研究问题
   - 读取 `context/` 下相关的个人上下文
   - 扫描知识库已有内容，标记已知/未知
   - 列出搜索方向
   - 向用户确认研究方向，用户可补充/调整

2. **搜集**：搜索 + 抓取
   - web search 搜索相关资料
   - 用 webReader 抓取有价值的内容到 `materials/`
   - 从知识库中挑出已有的相关 sources/topics

3. **研究**：读材料 → 写笔记（可多轮循环 ②③）
   - 逐篇阅读 materials/
   - 在 `notes.md` 记录关键发现、交叉验证、信息缺口
   - 发现缺口时继续搜索补充
   - 可主动向用户汇报阶段性发现，听取反馈调整方向

4. **输出**：写 `report.md`
   - 综合结论
   - 标注信息来源（链接到 materials/ 或外部 URL）
   - 列出未解决问题

5. **讨论与补充**：报告输出后，用户继续讨论追问
   - 将有价值的讨论内容追加到 `notes.md` 的 `## 讨论记录` 部分
   - 按日期组织，格式：`### YYYY-MM-DD：主题`，记录核心观点和结论
   - 讨论产生新决策时，更新 `decisions/`
   - 发现新信息缺口时，更新 notes.md

6. **入库（可选）**：AI 判断产出类型
   - **知识型**（方法论、原理等长期知识）：关键发现 → 新建 `wiki/sources/` 知识卡片，更新 topics/、index.md、log.md
   - **决策型**（时效性分析、个人建议）：报告保留在 analysis/，同时在 `decisions/` 创建决策记录
   - 不确定时，向用户建议并确认："这是时效性分析，建议保留在 analysis/ 不入库，可以吗？"

7. **跟进（强制）**：研究完成后，AI 必须向用户确认跟进项
   - 从报告中找出需要持续跟踪的发现（如数据变化、政策动态、业务指标等）
   - 向用户确认："以下发现可能需要持续跟进：1. xxx 2. xxx，需要创建跟踪项吗？"
   - 用户确认后，在 `follow-ups/` 下创建跟踪文件
   - 同时在 `todos/active.md` 的 `## 跟踪项` 下创建课题子标题（如 `### 拼多多`），添加跟踪待办

### plan.md 格式

```markdown
---
topic: 课题名称
created: YYYY-MM-DD
status: active | completed
---

## 研究问题
要回答的核心问题。

## 已有认知
来自知识库的已有内容（引用 sources/）。

## 信息缺口
还不知道、需要搜索的方向。

## 进度
- [x] 已完成的步骤
- [ ] 下一步
```

```markdown
---
topic: 课题名称
created: YYYY-MM-DD
completed: YYYY-MM-DD
updated: YYYY-MM-DD
status: completed
---

## 结论

> **结论时间：YYYY-MM-DD** | 后续更新见 follow-ups/

对研究问题的回答。

## 核心发现
- 发现1（→ 来源）
- 发现2（→ 来源）

## 个人建议（基于 context/）
<!-- 如读取了个人上下文，在此给出针对性建议；否则省略此节 -->
<!-- ⚠️ 持仓建议不写具体比例，只写方向（加仓/减仓/维持/清仓），比例由用户自己决定 -->

## 详细分析
### 子问题1
...

### 子问题2
...

## 未解决问题
- ...

## 参考资料
- [资料1](materials/001-xxx.md)
- [资料2](materials/002-xxx-xxx.md)
- 外部链接...
```

**结论时间戳规则**：

1. **结论必须标注时间**：每次更新结论时，在结论开头标注 `> **结论时间：YYYY-MM-DD**`，让读者知道这个判断是什么时候做的
2. **事件必须标注日期**：结论中提到的任何事件（财报、政策、数据发布等）必须附带具体日期（如"Q2 FY26财报（2026-04-29）"），不能只写"最新财报"
3. **结论过时需更新或标注**：如果结论中的关键数据已超过时效窗口（见数据时效性规则），要么更新结论，要么在结论中标注 `[⚠️ 数据截至 YYYY-MM-DD，距今 X 个月]`
4. **更新结论不删除旧判断**：如果结论随时间变化，保留判断演变过程（用日期标注），不要覆盖历史判断

### follow-up 格式（follow-ups/）

文件名：`{跟踪项简称}.md`

**普通跟踪项**：

```markdown
---
topic: 跟踪项名称
parent: analysis/{课题名}
cadence: daily | weekly | monthly
status: active | closed
created: YYYY-MM-DD
---

## 为什么要跟踪
研究报告中的哪个发现触发了这个跟踪项。

- **YYYY-MM-DD**: 初始状态——xxx，来源 [xxx](../materials/001.md)

```

**决策型跟踪项（macro-tracker）**：

当课题涉及**多个相互影响的变量**时（几乎所有宏观和公司研究都适用），创建此类型跟踪项。核心价值：将散装指标串联成变量网络，定义事件日历和确认规则，使跟踪具有预判能力。

```markdown
---
topic: 跟踪主题名称
parent: analysis/{课题名}
cadence: weekly
status: active
type: macro-tracker
created: YYYY-MM-DD
---

## 变量地图

用箭头连接变量间的传导关系，展示谁影响谁。

示例（美债利率）：
```
油价(霍尔木兹) → CPI/PPI → Fed预期(降/加息) → 收益率(10Y/30Y) → 持仓估值
                    ↘ 国债拍卖(bid-to-cover) ↗
```

## 变量分层

将变量分为两层：**当前走势驱动**（正在影响股价的短期变量）和**长期结构关键**（决定价值走向的核心变量）。这是 macro-tracker 最实用的部分——帮助判断"现在该看什么"和"长期该等什么"。

### 当前走势驱动（短期，1-4周）

正在主导股价走势的变量，通常与近期事件/数据/情绪直接关联。

| 变量 | 影响权重 | 当前状态 | 传导路径 | 下一观察点 |
|------|---------|---------|---------|-----------|
| 变量A | ★★★ | 具体状态 | A → B → 股价 | 什么事件/日期 |

### 长期结构关键（季度-年）

决定公司/资产长期价值走向的核心变量，短期不显眼但最终决定胜负。

| 变量 | 结构性/周期性 | 当前状态 | 转折信号 | 长期影响 |
|------|-------------|---------|---------|---------|
| 变量X | 结构性 | 具体状态 | 什么变化意味着趋势反转 | 对估值/基本面影响 |

## 关键变量与当前值

| 变量 | 当前值 | 上期 | 趋势 | 数据来源 |
|------|--------|------|------|---------|
| 变量1 | xxx | xxx | ↑↓→ | 来源 |
| 变量2 | xxx | xxx | ↑↓→ | 来源 |

## 事件日历

| 日期 | 事件 | 预期/前值 | 实际 | 对变量的影响 |
|------|------|----------|------|------------|
| YYYY-MM-DD | 事件名称 | 预期值 | — | 影响哪些变量 |
| YYYY-MM-DD | 事件名称 | — | — | — |

注：日历包含可预知的定期事件（CPI发布、财报、FOMC等）和已知的催化剂日期。

## 确认规则

定义几个信号同时出现时才确认趋势/触发操作：

- **趋势A确认**：条件1 + 条件2 → 判断/操作
- **趋势B确认**：条件1 + 条件2 → 判断/操作
- **反转信号**：条件 → 趋势可能改变

## 链条复盘

当变量发生显著变化时，记录事件链条（单事件→确认→发酵→新共识）：

- **YYYY-MM-DD ~ YYYY-MM-DD**: 起因→传导→结果
  - 模式：描述累积过程
  - 耗时：X天从A到B

## 跟踪记录

- **YYYY-MM-DD**: 初始状态——xxx，来源 [xxx](../materials/001.md)
- **YYYY-MM-DD**: 更新——xxx（标注哪些变量变了、是否触发确认规则）
```

## 搜索时效性规则

AI 搜索信息时必须遵守以下规则，确保数据时效性：

1. **搜索词必须包含当前年份**：所有搜索查询必须包含当前年份（如 2026），避免返回过期数据
2. **中英文双语搜索**：同一主题至少用中英文各搜一次（如"高通财报"和"Qualcomm earnings 2026"），覆盖不同信息源
3. **变量分层搜索**：决策型研究至少覆盖以下层次，每层至少2个搜索词：
   - 基本面变量（营收、利润、指引）
   - 估值与技术面变量（P/E、目标价、关键价位）
   - 行业/竞争变量（市场份额、竞品动态）
   - 宏观变量（利率、通胀、油价）
   - 政治/监管变量（政策、制裁、监管）
   - 情绪/资金面变量（Reddit、机构持仓、内部人交易）
4. **链条式发现**：搜索过程中发现新变量时，必须追加搜索该变量的细节
5. **数据标注"截至"日期**：引用任何量化数据必须标注数据截至日期

## 个人上下文（context/）

`context/` 存放用户的个人资料，供 AI 在研究和问答时作为决策参考。与知识库（关于世界的知识）不同，context 是**关于用户自身的信息**。

- **维护者**：用户自行维护，AI 不修改内容
- **读取时机**：研究或问答涉及个人决策时，AI 读取相关上下文文件
- **识别规则**：AI 根据研究主题自动判断需要读取哪些 context 文件（如涉及投资决策则读 `finance.md`）
- **过期提醒**：AI 发现上下文信息可能过时时，可提醒用户更新

## 决策记录（decisions/）

记录用户的重要决策及后续复盘，用于定期回顾和反思。

### 决策来源

- **研究产出**：决策型研究完成后，自动在 `decisions/` 创建记录
- **用户口述**：用户说"记录一个决策"时手动创建

### 决策记录格式

文件名：`YYYY-MM-DD-简短标识.md`

```markdown
---
title: 决策标题
date: YYYY-MM-DD
tags: [投资/职业/生活]
status: open
review_date: YYYY-MM-DD
context: analysis/xxx   # 关联的研究（如有）
---

## 决策
做了什么决定。

## 当时理由
为什么这样做。

## 预期
期待什么结果。

## 复盘
（待复盘）
```

### 状态流转

- `open` → 待复盘（review_date 到期后由复盘流程处理）
- `reviewed` → 已复盘
- `closed` → 决策已结束，不再跟进

### 复盘流程

由 `/wiki-review` skill 执行，可手动触发或由系统定时任务触发。扫描到期的 open 状态决策，逐条与用户回顾实际结果，记录教训。

## 问答规则

用户提问时触发 `/wiki-ask` skill。核心流程：

1. 先读 `wiki/index.md` 了解知识库全貌
2. 提取问题关键词，两轮搜索定位相关文件：
   - 第一轮：在 sources/topics/analysis/decisions/ 中按标签和标题匹配
   - 第二轮：精读候选文件判断相关性，不足时沿 `related` 扩展
3. 基于知识库内容回答，标注引用来源
4. 涉及个人决策时读取 `context/` 下相关文件
5. 知识库中没有相关内容时如实告知，可建议"研究 XX"补充

## 标签体系

AI 自动打标签，不预定义。常见维度：
- 领域：技术、股票、生活、健康、读书、工作
- 具体主题：Rust、新能源、量化交易
- 类型：教程、观点、研报、经验

## 信息来源可信度评估

研究过程中收集的所有信息必须按可信度分层标注，并在 notes.md 中记录评估。

### 分层标准

| 层级 | 来源类型 | 特征 | 用法 |
|------|---------|------|------|
| **高** | 官方文件（SEC 财报、10-K/10-Q、6-K）、法定披露（13F）、监管文件 | 法律义务披露，造假面临处罚 | 直接作为判断依据 |
| **中** | 商业数据商（Sensor Tower、Statista）、投行研报、权威财经媒体 | 方法论可追溯但假设多，数值有偏差（±15-20%） | 方向判断，不依赖精确数字 |
| **低** | 社区匿名帖（Reddit、论坛）、自媒体、二手转述、无法验证来源 | 无法验证身份/数据出处，存在利益相关或选择性披露 | 标记为风险假设，触发进一步验证，不直接用于决策 |

### 评估维度

1. **可验证性**：是否有独立来源可以交叉验证？
2. **利益相关**：信息发布者是否有立场偏向？（如满仓者写看多帖）
3. **信息链**：一手数据还是 N 手转述？原始出处是什么？
4. **时效性**：数据是否已过时？

### 记录要求

- 在 notes.md 的每个信息来源旁标注可信度（高/中/低）
- 研究报告中引用低可信度数据时，必须注明"未经验证"或"需独立验证"
- 交叉验证时记录哪些来源互相印证

## 数据时效性规则

AI 必须对数据的时效性保持清醒认知。旧数据本身不是问题（回溯分析很有价值），问题在于**不知道数据是旧的，错当最新状态使用**。

### 核心原则

1. **每个量化数据点必须标注"截至 YYYY-MM-DD"**，写入时以当前日期为锚点计算距今时长
2. **区分用途**：回溯分析（"当时是 X"）vs 现状判断（"目前是 X"），两者不能混用
3. **过时数据不能直接用于现状判断**：如果数据超过时效窗口（见下表），用于现状分析时必须搜索最新数据补充

### 时效窗口

| 数据类型 | 时效窗口 | 示例 |
|---------|---------|------|
| 市场价格、行情 | 日/周 | 股价、油价、汇率 |
| 经济指标 | 月度/季度 | CPI、GDP、非农、PMI |
| 公司财务数据 | 季度 | 营收、利润、指引 |
| 行业趋势 | 月度 | 市场份额、渗透率、出货量 |
| 政策法规 | 视具体事件 | 关税、监管、贸易协议 |
| 方法论、原理 | 长期有效 | 投资框架、技术原理、历史规律 |

### 在研究流程中的应用

- **搜集材料时**：提取每个材料的数据截至日期，写入 frontmatter `data_as_of` 字段
- **分析时**：notes.md 中对过时数据标注 `[⚠️ 数据截至 YYYY-MM-DD，距今 X 个月]`
- **报告中**：核心发现中的关键数据标注时效，如"2Y-10Y 利差 +50bps（截至 2025-07，距今 10 个月，当前值待查）"
- **过时数据触发更新**：核心数据超过时效窗口时，主动搜索最新数据，补充变化趋势

## 跨课题关联（connections/）

知识库的核心差异点：单篇文章内容搜索引擎都有，**不同信息交叉分析产生的新洞察**才是独特的。connections 记录跨课题的关联发现，每张卡片产出单篇研究得不到的新判断。

### 与 topics 的关系

- **topics/** = 专题内聚合（同标签文章汇总，信息整理）
- **connections/** = 专题间关联（跨课题交叉分析，A + B → C，信息创造）

### 关联类型

| 类型 | 含义 |
|------|------|
| causal | 因果链（A 导致/影响 B） |
| resonance | 共振（不同领域同步变化，共享深层驱动） |
| contradiction | 矛盾（不同课题结论冲突） |
| hedge | 对冲（A 利好 = B 利空） |

### 关联分析卡片格式

文件名：`wiki/connections/YYYY-MM-DD-简短标题.md`

```markdown
---
title: 关联标题
created: YYYY-MM-DD
type: causal | resonance | contradiction | hedge
topics: [课题1, 课题2]
sources: [相关文件路径]
---

## 关联发现
A 的发现 + B 的发现 → 新洞察 C（一句话概括）

## 具体分析
- 课题A：相关发现
- 课题B：相关发现
- 交叉推导：从 A+B 推导出的新判断

## 对现有认知的影响
- 课题A：...
- 课题B：...
```

### 触发方式

嵌入已有流程，不单独触发：
- **研究过程**（深度）：wiki-research 分析完成后检查核心发现是否与其他课题有关联
- **文章入库**（轻量）：wiki-digest 入库新文章时检查跨专题语义关联
- **Sweep 扫描**（兜底）：维度 K 定期扫描未记录的跨课题关联

## 必须使用中文

所有生成的文件内容使用中文，文件名可包含英文。

## 定时任务通知（pending.md 邮箱）

`.cron/pending.md` 是定时任务的通知邮箱。cron 任务在无人值守时发现问题，写入此文件；AI 在每次对话时检查并提醒用户。

### 通知流程

```
cron 执行 → 发现问题 → 写入 .cron/pending.md
                            ↓
用户对话 → AI 检查 .cron/pending.md → 有内容 → 回复末尾附上提醒
                            ↓
用户确认"知道了" → 清空 pending.md
```

### AI 行为规则

**每次回复用户时**：
1. 检查 `.cron/pending.md` 是否存在且非空
2. 有内容 → 在回复末尾用分隔线附上提醒
3. 用户确认已知晓后，清空该文件
4. 不存在或为空 → 不输出任何提醒
5. 检查 `.cron/sweep-issues.md` 是否有未解决问题（`- [ ]`）
6. 有未解决问题 → 在 pending.md 提醒之后附上："知识库有 N 个未解决的扫描问题，说'扫一下'处理"

### 通知格式

```
---
⚡ 定时扫描提醒：{pending.md 中的内容，一行一条}
```

### 写入者

| 任务 | 通知内容 |
|------|---------|
| wiki-digest | 处理了 N 篇文章（标题列表） |
| wiki-sweep | 发现的遗漏跟进项 |
| wiki-review | 到期需要复盘的决策 |
| todo-remind | 待办数统计（早晚汇总）+ 定时提醒（带 remind 标记的待办） |
