# Research — 课题研究流程

## 触发条件

- 用户说"研究"、"研"、"调研"、"了解"、"深入看看"，并带有明确主题
- 示例："研究成交量择时因子"、"研拼多多"、"调研 Rust 异步运行时"

**缺少主题时**：回复"研究什么主题？"并停止执行。

## 前置检查

1. **获取准确时间**（必须）：执行 `date "+%Y-%m-%d %H:%M %a"` 获取当前日期、时间和星期，作为全流程的时间锚点。所有数据时效性判断、时间相对计算都基于此时间
2. 检查环境变量 `WIKI_ROOT` 是否设置。未设置则报错："未设置 WIKI_ROOT 环境变量，无法定位知识库。请运行 `echo $WIKI_ROOT` 检查。"
2. 检查 `$WIKI_ROOT/inbox/`、`$WIKI_ROOT/raw/`、`$WIKI_ROOT/wiki/` 三个目录是否存在
3. 检查 `$WIKI_ROOT/wiki/analysis/` 是否存在（不存在则创建）
4. 检查 `$WIKI_ROOT/context/` 是否存在（如存在，列出文件供后续按需读取）
5. 检查 `$WIKI_ROOT/decisions/` 是否存在（不存在则创建）
6. 不满足时报错并提示检查 WIKI_ROOT 配置

## 执行步骤

### 1. 开题

1. 读取 `wiki/index.md` 了解知识库全貌
2. 检查 `context/` 下是否有相关个人上下文（如涉及投资读 `finance.md`），纳入研究背景
3. 扫描 `wiki/sources/` 和 `wiki/topics/` 下已有相关内容，标记已知/未知
4. 创建 `wiki/analysis/{课题slug}/plan.md`（slug 规则：英文小写+短横线，中文保留）

plan.md 格式：
```markdown
---
topic: 课题名称
created: YYYY-MM-DD
status: active
type: book  <!-- 仅书籍课题添加此字段 -->
distill: true  <!-- 仅用户确认蒸馏时添加此字段 -->
---

## 研究问题
要回答的核心问题（1-3个）。

## 已有认知
来自知识库的已有内容（引用 sources/），如无则写"知识库暂无相关内容"。

## 信息缺口
还不知道、需要搜索的方向。

## 搜索计划
- 方向1：具体搜索关键词和目标
- 方向2：...

## 进度
- [x] 开题
- [ ] 搜集资料
- [ ] 研究分析
- [ ] 输出报告
- [ ] 知识入库
```

**书籍检测**：以下任一条件满足时，识别为书籍课题：
- 课题名称包含书名号《》
- 用户明确说这是一本书的研究（如"研究一下这本书"）
- **AI 判断课题名称可能是书名**：课题名称听起来像书名（如"XX研究"、"XX指南"、"XX与XX"、特定知名作品名），在开题阶段**主动询问**用户："这个课题是一本书吗？还是普通主题研究？"

识别为书籍后，在 plan.md 中添加 `type: book`，然后**依次处理**：

1. **书籍文件**：
   - 用户已提供路径 → 验证存在，记录到 plan.md
   - 用户要求下载 → 搜索书名 + PDF/电子书/下载，存入 `wiki-engine/data/{书名slug}.pdf`，记录路径
   - 用户未提及 → 询问：
     ```
     请提供书籍文件：
     - 书籍路径（如 data/xxx.pdf 或本地任意路径）
     - 如果没有文件，我可以尝试搜索下载 PDF
     ```

2. **蒸馏意愿**：使用 `cangjie-skill`（book2skill）的 RIA-TV++ 流程蒸馏方法论
   - 用户已明确要求蒸馏（如"提炼skill"、"蒸馏"、"输出skill"等表述）→ 直接设置 `distill: true`，告知用户："将使用 cangjie-skill 蒸馏方法论为可执行 skill，产出存放在 skills/ 目录并通过软链接接入全局 CC。"
   - 用户未提及 → 询问：
     ```
     是否同时蒸馏方法论为 skill？
     使用 cangjie-skill（RIA-TV++ 流程）从书中提取可执行方法论。
     - 是：研究 + 蒸馏一气呵成，产出知识层（report）+ 工具层（skills/）
     - 否：只做常规研究，产出知识层
     ```
   - 用户说否 → 不蒸馏

**书籍文件处理**：
- 用户提供路径 → 验证文件存在，将路径记录在 plan.md 的 `## 书籍文件` 节
- 用户无文件 → 搜索书名 + PDF/电子书/下载，找到后存入 `wiki-engine/data/{书名slug}.pdf`，记录路径
- 支持的格式：PDF（用 Read 工具读取）、txt/md（直接读取）
- 如果文件放在 `wiki-engine/data/` 下，不会被 git 提交（.gitignore 已排除内容）

**向用户展示 plan.md 的核心内容**（研究问题、已有认知、信息缺口），确认研究方向。用户可以补充/调整/直接确认。

### 2. 搜集资料

基于 plan.md 的搜索计划搜集：

**已有 skill 检索**（书籍课题执行）：如果 plan.md 标记了 `type: book`，检查 `~/.claude/skills/books/_index.md` 是否有与当前课题相关的方法论 skill。如有，在 plan.md "已有认知"中列出这些 skill 的名称和摘要，供研究阶段参考。

**书籍内容读取**（书籍课题执行）：如果 plan.md 标记了 `type: book` 且有书籍文件路径：
1. 读取书籍全文（PDF 用 Read 工具，按章节分批读取）
2. 将各章节内容保存到 `materials/` 下：`000-章节名.md`（编号从 000 起，区别于后续搜索材料）
3. 每个章节文件的 frontmatter：
   ```yaml
   title: 章节名
   source: book
   book: 书名
   fetched: YYYY-MM-DD
   ```
4. 书籍内容是**核心材料**，后续 web 搜索是**补充**（书评、作者背景、延伸讨论等）

**搜索策略**：
- 每个方向用 WebSearch 搜索，关键词从搜索计划提取
- 多个独立方向可并行（使用 Agent 工具）
- 每轮搜索 3-5 个查询，优先中文内容（除非用户指定英文源）

**搜索时效性规则**（必须遵守）：
- 所有搜索词必须包含当前年份（从前置检查获取），避免返回过期数据
- 中英文双语搜索：同一主题至少用中英文各搜一次
- 链条式发现：搜索中发现新变量时，追加搜索该变量的细节

**分层搜索检查清单**（决策型研究必须逐层覆盖）：

对于涉及投资/决策的研究课题，必须按以下9层逐层搜索，每层至少2个搜索词：

| 层次 | 搜索方向 | 搜索词示例 |
|------|---------|-----------|
| 1. 基本面 | 营收、利润、指引、分业务数据 | "[公司] earnings [YYYY]"、"[公司] revenue breakdown" |
| 2. 估值与技术面 | 估值倍数、分析师目标价、关键价位 | "[公司] target price [YYYY]"、"[公司] P/E ratio" |
| 3. 行业/竞争 | 竞品动态、市场份额、行业趋势 | "[行业] market share [YYYY]"、"[竞品] vs [公司]" |
| 4. 宏观 | 利率、通胀、油价、GDP | "US Treasury yield [YYYY]"、"CPI [YYYY]" |
| 5. 政治/监管 | 贸易政策、制裁、监管变化 | "[公司] regulation [YYYY]"、"US China trade [YYYY]" |
| 6. 情绪/资金面 | Reddit讨论、机构持仓、内部人交易 | "[公司] Reddit [YYYY]"、"[公司] institutional holdings" |
| 7. 产业链上下游 | 供应商/客户/合作伙伴动态 | "[供应商名] [公司]"、"[公司] supply chain [YYYY]" |
| 8. 跨领域交叉 | 标的+半导体/AI/芯片等跨界词 | "[公司] chip"、"Musk semiconductor"、"郭明錤 [公司]" |
| 9. 行业分析师 | 知名分析师+标的组合 | "郭明錤 [公司]"、"Dan Ives [公司]"、"Ming-Chi Kuo [company]" |

**补充搜索层**（每轮搜集必须执行）：
- **中文源扫描**：至少用中文搜一次"标的+最新/动态/新闻"，覆盖华尔街见闻、富途资讯、36氪、雪球
- **SEC/监管文件**：涉及美国上市公司时，检查 SEC EDGAR 最新 8-K/13F/SC 13D 等文件
- **叙事变量扫描**：除按计划搜索外，快速搜索是否有新的叙事变量出现（新项目、新合作、新诉讼等）

搜索总轮次不少于4轮，每轮至少2-3个搜索词，总搜索不少于10次。

**抓取规则**：
- 搜索结果中有价值的页面，用 `mcp__web_reader__webReader` 抓取全文
- 保存到 `wiki/analysis/{课题slug}/materials/`，文件名 `001-简短标题.md`、`002-...`
- 每个文件 frontmatter：`url`、`title`、`fetched`、`data_as_of`（材料中核心数据的截至日期）
- 抓取失败的 URL 跳过，在 notes.md 记录失败原因

**数据时效性标注**（必须）：
- 阅读抓取内容时，提取核心量化数据的截至日期（如"截至2025年7月"、"2025 Q3"等）
- 在 frontmatter 中写入 `data_as_of: YYYY-MM-DD`（无明确截至日期则写 `fetched` 的日期）
- 在材料正文中，关键数据旁标注 `[截至 YYYY-MM-DD]`
- 与前置检查获取的当前时间对比，超过时效窗口的标注 `⚠️ 距今 X 个月`（参见 CLAUDE.md 时效窗口表）

**知识库交叉**：同时从 `wiki/sources/` 找已有相关文章，在 notes.md 引用。

### 3. 研究分析

创建 `wiki/analysis/{课题slug}/notes.md`，逐篇阅读 materials/：

- 记录关键发现（标注来源 material 编号）
- 交叉验证不同来源
- 标记信息缺口
- **数据时效性检查**：对每个核心数据点，检查其 `data_as_of` 是否超过时效窗口（参见 CLAUDE.md）。过时数据标注 `⚠️ 数据截至 YYYY-MM-DD，距今 X 个月`，明确区分"回溯素材"和"当前状态"
- **过时数据触发更新**：核心数据超过时效窗口时，在下一轮搜索中优先搜索该数据的最新值，补充从旧到新的变化趋势
- **信息冲突处理**：不同来源说法矛盾时，在 notes.md 标注 `[冲突]`，记录各方来源和依据，向用户汇报让用户判断

**多轮循环**：发现缺口 → 回到步骤 2 继续搜索 → 补充到 materials/ → 继续分析
- 最多 3 轮，避免无限搜索
- 每轮完成后向用户简要汇报发现

**跨课题关联发现**（分析完成后执行）：
1. 回顾核心发现，检查是否与 `wiki/analysis/` 下其他课题或 `wiki/topics/` 下其他专题有关联
2. 关联类型：因果链、共振、矛盾、对冲（参见 CLAUDE.md「跨课题关联」）
3. 发现关联时记录下来，在报告的「跨课题关联」节中输出，并创建关联分析卡片到 `wiki/connections/`

**用户反馈点**：
- "继续深入 XX" → 补充搜索
- "差不多了，出报告" → 进入步骤 4
- 不回复 → 继续下一轮

### 4. 输出报告

创建 `wiki/analysis/{课题slug}/report.md`。

> ⚠️ **以下结构是固定的，不能增加、删除、重排章节。**
> 每次输出必须严格遵循此模板，不得自由发挥章节结构。

#### report.md 格式（强制）

```markdown
---
topic: 课题名称
created: YYYY-MM-DD
completed: YYYY-MM-DD
updated: YYYY-MM-DD
status: completed
---

## 结论

> **结论时间：YYYY-MM-DD**

对研究问题的直接回答（2-5 句话）。只放结论本身，不放补充分析、新发现、待验证变量。

## 核心发现
- 发现1（→ 来源：materials/001-xxx.md，数据截至 YYYY-MM-DD）
- 发现2（→ 来源：materials/002-xxx.md）
- 关键量化数据标注时效：如"2Y-10Y 利差 +50bps（截至 2025-07）"

## 持仓建议
<!-- 以下仅在决策型研究（涉及个人持仓/操作）时出现，知识型研究省略此节 -->

**操作方向**：加仓 / 减仓 / 维持 / 清仓 / 观望（不建仓）

**判断依据**：
- 1-2 句核心理由

**情景分析**：
| 情景 | 概率 | 条件 | 操作 |
|------|------|------|------|
| 乐观 | X% | 具体条件 | 具体操作 |
| 中性 | X% | 具体条件 | 具体操作 |
| 悲观 | X% | 具体条件 | 具体操作 |

**止损线**：跌破 $XX → 减仓/清仓（仅已有持仓时填写）

**加仓信号**：（以下条件满足 2 个以上时）
- 条件1
- 条件2

**减仓信号**：（以下任一触发时）
- 条件1
- 条件2

## 详细分析
### 子问题1
...

### 子问题2
...

## 未解决问题
- ...

## 跨课题关联
<!-- 有关联时才出现此节 -->
- 与「课题A」的关联：具体发现 → 交叉洞察
- 示例：WTI 涨至 $102+ → TEMU 跨境物流成本上升 → PDD 毛利率安全垫比单独评估更薄

## 数据时效性说明
| 核心数据 | 截至 | 距今 | 状态 |
|---------|------|------|------|
| 2Y-10Y 利差 | 2025-07 | 10个月 | ⚠️ 建议更新 |
| CPI 3月数据 | 2026-03 | 2个月 | ✅ 可用 |

## 参考资料
- [资料标题1](materials/001-xxx.md) - 一句话概括
- [资料标题2](materials/002-xxx.md) - 一句话概括
- 外部链接...
```

#### 格式禁止事项

- **禁止在"结论"节中混入"新发现""待验证变量""持仓建议"等内容**——各归其位
- **禁止自定义章节结构**（如"看空/看多""投资论点""多空辩论"）——统一用"详细分析"下的"子问题"分节
- **禁止省略"未解决问题"和"参考资料"节**（可以为空列表，但不能没有）
- **禁止使用 `# 标题` 开头替代 YAML frontmatter**
- **禁止"持仓建议"只写模糊表述**（如"关注""警惕""注意"）而不给明确操作方向（加仓/减仓/维持/清仓/观望）

#### 补充分析的正确放置

- 研究完成后的新发现/新变量 → "详细分析"中新增 `### 待验证：XXX` 子节
- 情景分析/操作建议 → "持仓建议"节
- 止损线/触发条件 → "持仓建议"节
- 方法论/原理等通用知识 → "详细分析"中对应子节
- 以上内容通过 follow-ups/ 跟踪，不混入结论

#### 输出前自检（必须逐条确认）

- [ ] frontmatter 包含 topic/created/completed/updated/status 五个字段
- [ ] 结论节有 `> **结论时间：YYYY-MM-DD**` 标注
- [ ] 结论节只有结论（2-5 句），没有补充分析、待验证变量等额外内容
- [ ] 核心发现是列表格式，每条附来源和时效标注
- [ ] 详细分析按子问题分节（不是按多空/投资论点等自定义结构）
- [ ] 决策型研究有"持仓建议"节，且包含明确操作方向（加仓/减仓/维持/清仓/观望）
- [ ] "持仓建议"包含情景分析表（概率+条件+操作）
- [ ] 有"未解决问题"节
- [ ] 有"参考资料"节
- [ ] 所有量化数据标注"截至 YYYY-MM-DD"

更新 plan.md 状态为 `completed`。

向用户展示**结论、持仓建议和核心发现**（不展示完整报告）。

### 5. 讨论与补充

报告输出后，用户可能继续追问和讨论：

- 有价值的讨论追加到 `notes.md` 的 `## 讨论记录`，按日期：`### YYYY-MM-DD：主题`
- 讨论产生新决策 → 更新 `decisions/`
- 发现新信息缺口 → 更新 notes.md

### 6. 知识入库 & 决策联动

#### 6a. 判断产出类型

**知识型**（方法论、原理、通用框架）：
1. 创建知识卡片到 `wiki/sources/`，`source: research`，`origin: analysis/{slug}/report.md`
2. 更新相关 topic，达到 3 篇时创建新 topic
3. 更新 `wiki/index.md` 和 `wiki/log.md`

**决策型**（时效性分析、个股研究、个人建议）：
- 报告保留在 `analysis/`，不入库
- 写入 `wiki/log.md`
- **必须执行 6b 决策联动**

**判断规则**：
- 特定个股/行业当前状态 → 决策型
- 通用方法论/原理/框架 → 知识型
- 不确定时问用户："这是时效性分析，建议保留在 analysis/ 不入库，可以吗？"

#### 6b. 决策联动（决策型研究强制执行）

当研究涉及个人持仓/操作建议时（即 report 包含"持仓建议"节），**必须**在 `decisions/` 下创建或更新决策记录。

**新建决策记录**（无同名标的 open 决策时）：

文件名：`YYYY-MM-DD-{标的简称}持仓决策.md`

```markdown
---
title: {标的}持仓决策
date: YYYY-MM-DD
tags: [投资]
status: open
review_date: YYYY-MM-DD（默认 3 个月后）
context: analysis/{课题名}
---

## 决策
{从 report 的"持仓建议"节提取操作方向和止损线}

## 当时理由
{从 report 的"核心发现"节提取 2-3 条关键依据}

## 预期
{从 report 的"持仓建议"节的情景分析提取}

## 复盘
（待复盘）
```

**更新已有决策**（decisions/ 下已有同一标的的 open 状态决策时）：
- 不创建新文件，而是在"复盘"节前添加 `## 决策更新 YYYY-MM-DD`
- 包含：新操作方向 + 变更理由（什么变了导致决策改变）
- 更新 `review_date` 为新日期 + 3 个月

创建/更新后向用户确认决策记录内容。

### 7. 跟进（强制，不可跳过）

研究完成后 **必须** 执行：

**7a. 变量梳理（决策型研究必须，知识型跳过）**

从研究材料中系统提取影响结论的所有变量，梳理传导关系：

1. **提取变量**：回顾 materials/ 和 notes.md，列出所有影响结论的变量（基本面、技术面、消息面、宏观、政治、情绪）
2. **画传导关系**：用箭头连接变量间的因果关系（谁影响谁、谁被谁影响）
3. **分层标注**：将变量分为"当前走势驱动"（1-4周内主导股价的变量）和"长期结构关键"（季度-年级别的核心变量），标注影响权重和转折信号
4. **标出关键事件**：哪些事件会改变哪些变量（财报日期、CPI发布、FOMC、产品发布等）
5. **定义确认规则**：几个信号同时出现才确认趋势/触发操作（避免单次数据就做判断）

将以上内容整合为一个 `macro-tracker` 类型的 follow-up（格式见 CLAUDE.md「决策型跟踪项」）。

**7b. 创建跟踪项**

1. 回顾 report.md 的核心发现和未解决问题
2. 识别需要持续跟踪的项（数据变化、业务指标、政策动态、技术演进）
3. 向用户确认：
   ```
   以下发现可能需要持续跟进：
   1. xxx（如：PDD 半托管模式占比变化）
   2. xxx

   同时，我梳理了影响结论的关键变量和传导关系，建议创建一个变量跟踪项（macro-tracker）。
   需要创建跟踪项吗？可以选择全部或指定编号。
   ```
4. 用户确认后：
   - 在 `follow-ups/` 下创建跟踪文件（格式见 CLAUDE.md）
   - 决策型课题：创建1个 macro-tracker（变量地图+事件日历+确认规则），替代散装的多个独立 follow-up
   - 在 `todos/active.md` 对应分类的 `### 长期` 下添加：`- [ ] 跟进：{课题} - {项} \`cadence: monthly\` \`created: YYYY-MM-DD\``
5. 用户说不需要 → 仅记录到 log.md

### 8. 蒸馏（仅书籍课题 + 用户确认时执行）

当 plan.md 同时标记 `type: book` 和 `distill: true` 时，在研究完成（步骤 4-7）后执行蒸馏。

#### 8a. 调用 cangjie-skill

使用 `cangjie-skill`（book2skill）的流程，从研究材料中蒸馏可执行方法论：

1. 将 materials/ 和 notes.md 作为输入，调用 cangjie-skill 的提取流程
2. 输出目录设为 `wiki/analysis/{课题slug}/skills/`（而非 cangjie-skill 默认的 `books/` 目录）
3. cangjie-skill 产出：
   - `skills/INDEX.md` — 候选 skill 索引
   - `skills/candidates/` — 待筛选的 skill 候选
   - `skills/rejected/` — 不达标的候选
   - `skills/{skill名}/SKILL.md` — 最终产出的 skill
   - `skills/{skill名}/test-prompts.json` — 测试提示词

#### 8b. 创建全局软链接

蒸馏完成后，将产出的 skill 通过软链接接入全局 CC：

1. 确保 `~/.claude/skills/books/` 目录存在
2. 为每个产出的 skill 创建软链接：
   ```bash
   ln -sf "$WIKI_ROOT/wiki/analysis/{课题slug}/skills/{skill名}/SKILL.md" \
          ~/.claude/skills/books/{课题名}-{skill名}.md
   ```
3. 更新 `~/.claude/skills/books/_index.md`，添加新条目：
   ```markdown
   - [{课题名} - {skill名}](books/{课题名}-{skill名}.md) — 一句话摘要
   ```

4. 向用户汇报蒸馏结果："从《{书名}》中蒸馏出 N 个方法论 skill：{skill名列表}，已链接到全局 CC 可复用。"

- **搜索无结果**：告知用户，建议调整关键词或方向
- **网页抓取失败**：跳过，在 notes.md 记录失败 URL，继续其他来源
- **课题目录已存在**：检查 plan.md status，active → 问是否继续；completed → 问是否重新研究
- **信息严重不足**：诚实告知已获取信息和剩余缺口，由用户决定是否继续
