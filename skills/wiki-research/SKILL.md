# Research — 课题研究流程

## 触发条件

- 用户说"研究"、"调研"、"了解"、"深入看看"等指令，并带有明确的研究主题
- 示例："研究成交量择时因子"、"调研 Rust 异步运行时"、"帮我了解 EOM 指标"

**缺少主题时：** 用户只说"研究"但没有说明研究什么，回复"研究什么主题？" 并停止执行。

## 前置检查

1. 检查当前工作目录是否符合 wiki 规范：需存在 `inbox/`、`raw/`、`wiki/` 三个目录
2. 检查 `wiki/analysis/` 目录是否存在（如不存在则创建）
3. 检查 `context/` 目录是否存在，如存在则扫描其中的文件列表，供后续按需读取
4. 如果不满足，报错并提示用户切换到 wiki 根目录

所有文件操作均基于当前工作目录，使用相对路径。

## 执行步骤

### 1. 开题

读取 `wiki/index.md` 了解知识库全貌，然后创建研究工作区：

**读取个人上下文：** 检查 `context/` 目录下是否有与研究主题相关的个人上下文文件（如涉及投资则读 `finance.md`），将其内容纳入研究背景。

```
wiki/analysis/{课题slug}/plan.md
```

plan.md 内容：

```markdown
---
topic: 课题名称
created: YYYY-MM-DD
status: active
---

## 研究问题
要回答的核心问题（1-3个）。

## 已有认知
来自知识库的已有内容（引用 sources/），如无则写"知识库暂无相关内容"。

## 信息缺口
还不知道、需要搜索的方向。

## 搜索计划
- 方向1：...
- 方向2：...

## 进度
- [x] 开题
- [ ] 搜集资料
- [ ] 研究分析
- [ ] 输出报告
- [ ] 知识入库
```

**向用户展示 plan.md 的核心内容**（研究问题、已有认知、信息缺口），确认研究方向。用户可以：
- 补充研究问题
- 调整搜索方向
- 确认后继续

### 2. 搜集资料

基于 plan.md 的搜索计划，使用 WebSearch 搜索相关资料。

对搜索到的有价值内容：
- 使用 `mcp__web_reader__webReader` 抓取全文到 `wiki/analysis/{课题slug}/materials/`
- 文件名：`001-简短标题.md`、`002-简短标题.md`...
- 每个 material 文件头部加 frontmatter：`url`、`title`、`fetched`

同时从知识库 `wiki/sources/` 中找出已有的相关文章，在 notes.md 中引用。

### 3. 研究分析

创建 `wiki/analysis/{课题slug}/notes.md`，逐篇阅读 materials/ 中的资料：

- 记录关键发现
- 交叉验证不同来源的信息
- 标记信息缺口（如发现新缺口，回到步骤 2 继续搜索）
- 形成对研究问题的初步回答

**多轮循环：** 发现信息缺口时，继续搜索补充材料（重复 2-3），直到：
- 研究问题得到充分回答，或
- 已搜索多轮仍无法找到更多信息

**阶段性汇报：** 每完成一轮搜集+分析，可向用户简要汇报发现，听取反馈调整方向。用户可以：
- 说"继续深入 XX 方向" → 补充搜索
- 说"差不多了，出报告" → 进入步骤 4
- 不回复 → 继续下一轮

### 4. 输出报告

创建 `wiki/analysis/{课题slug}/report.md`：

```markdown
---
topic: 课题名称
created: YYYY-MM-DD
completed: YYYY-MM-DD
status: completed
---

## 结论
对研究问题的直接回答（简洁明确）。

## 核心发现
- 发现1（→ 来源：materials/001-xxx.md）
- 发现2（→ 来源：materials/002-xxx.md）
- ...

## 个人建议（基于 context/）
<!-- 如读取了个人上下文，在此给出针对性建议；否则省略此节 -->

## 详细分析
### 子问题1
...

### 子问题2
...

## 未解决问题
- ...

## 参考资料
- [资料标题1](materials/001-xxx.md) - 一句话概括
- [资料标题2](materials/002-xxx.md) - 一句话概括
- 外部链接...
```

更新 plan.md 状态为 `completed`。

向用户展示报告的**结论和核心发现**，告知完整报告已保存。

### 5. 知识入库（可选）

研究完成后，AI 判断产出类型：

**知识型**（方法论、原理、通用框架等长期知识）：
1. 创建知识卡片到 `wiki/sources/`，`source` 字段标记为 `research`，`origin` 指向 `analysis/{课题slug}/report.md`
2. 更新相关 topic，达到 3 篇时创建新 topic
3. 更新 `wiki/index.md` 和 `wiki/log.md`

**决策型**（时效性分析、个股研究、个人建议等）：
- 报告保留在 `analysis/`，不入库
- 在 `decisions/` 下创建决策记录（`YYYY-MM-DD-简短标识.md`），包含决策内容、理由、预期、review_date（默认 3 个月后）
- 追加 `wiki/log.md`：
  ```
  - YYYY-MM-DD HH:mm: 完成课题研究「课题名称」→ analysis/{课题slug}/ + decisions/（决策型）
  ```

**判断规则**：
- 研究对象是特定个股/行业当前状态 → 决策型
- 研究对象是通用方法论/原理/框架 → 知识型
- 不确定时，向用户建议并确认："这是时效性分析，建议保留在 analysis/ 不入库，可以吗？"

## 用户交互规则

- **开题阶段**：必须向用户确认研究方向后再开始搜集
- **研究过程**：每轮搜集后简要汇报进展，不过度打扰
- **最终报告**：展示结论和核心发现，不展示完整报告全文
- **入库通知**：告知新增了哪些知识卡片

## 错误处理

- **搜索无结果**：告知用户，建议调整关键词或方向
- **网页抓取失败**：跳过该来源，在 notes.md 中记录失败 URL，继续其他来源
- **课题目录已存在**：检查 plan.md 的 status，如为 active 则询问是否继续上次研究；如为 completed 则询问是否重新研究
- **信息严重不足**：诚实告知用户，列出已获取的信息和剩余缺口，由用户决定是否继续

## 路径约定

所有路径基于当前工作目录（相对路径）。当前目录必须有 `inbox/`、`raw/`、`wiki/` 三个目录才算有效的 wiki 根目录。
