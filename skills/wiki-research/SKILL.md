# Research — 课题研究流程

> **分层**：① 输入类 · 内容进知识库
> **被调**：用户
> **调用**：wiki-mine（步骤 3 挖盲区变量）、wiki-lens（收尾步骤透镜碰撞）

## 触发条件

- 用户说"研究"、"研"、"调研"、"了解"、"深入看看"、"刷新"、"重新研究"、"更新一下XX专题"，并带有明确主题
- 示例："研究成交量择时因子"、"研拼多多"、"调研 Rust 异步运行时"、"刷新 AI 板块专题"、"重新研究拼多多"

**缺少主题时**：回复"研究什么主题？"并停止执行。

## 两种执行模式

本 skill 自动检测课题是否已存在，选择对应模式：

| 模式 | 触发条件 | 起点 | 深度 | 产出 |
|------|---------|------|------|------|
| **新课题模式** | `wiki/analysis/{slug}/` 不存在或无 report.md | 空白 | 完整研究 | 新 report.md + tracker |
| **刷新模式** | `wiki/analysis/{slug}/` 已存在且有 report.md 或 follow-ups/ | 已有 tracker + report | **等同新课题深度** | delta report + 更新 tracker |

**刷新模式核心定位**：用户最高频的需求——不是从零开始，但深度必须等同从零研究。已有 tracker 是起点不是约束，挖盲区 + 深度分析新盲区 + 更新旧变量 + 重新评估结论，一个流程搞定。

**模式选择规则**：
- 检测到课题目录已存在 → 默认进入刷新模式，但向用户确认
- 用户说"重新研究"或"从零开始" → 强制新课题模式（备份旧目录为 `{slug}-v1`）
- 用户说"刷新"、"更新"、"深度更新" → 强制刷新模式

🔴 **CHECKPOINT: 模式确认**
检测到课题已存在时，必须向用户确认：
```
检测到「{课题}」已有研究：
- 报告完成于 {date}（report.md）
- tracker 最后更新 {date}
- 跟踪记录 N 条

执行方式：
1. 刷新模式（推荐）：保留历史，在已有基础上做深度更新——挖盲区+深度分析+更新数据+复盘上次判断
2. 重新研究：备份旧目录为 {slug}-v1，从零开始
3. 取消
```
未获确认不要继续。

## 前置检查（两种模式共用）

1. **获取准确时间**（必须）：执行 `date "+%Y-%m-%d %H:%M %a"` 获取当前日期、时间和星期，作为全流程的时间锚点。所有数据时效性判断、时间相对计算都基于此时间
2. 检查环境变量 `WIKI_ROOT` 是否设置。未设置则报错："未设置 WIKI_ROOT 环境变量，无法定位知识库。请运行 `echo $WIKI_ROOT` 检查。"
3. 检查 `$WIKI_ROOT/inbox/`、`$WIKI_ROOT/raw/`、`$WIKI_ROOT/wiki/` 三个目录是否存在
4. 检查 `$WIKI_ROOT/wiki/analysis/` 是否存在（不存在则创建）
5. 检查 `$WIKI_ROOT/context/` 是否存在（如存在，列出文件供后续按需读取）
6. 检查 `$WIKI_ROOT/decisions/` 是否存在（不存在则创建）
7. 不满足时报错并提示检查 WIKI_ROOT 配置

## 执行步骤

### 步骤总览

```
新课题模式：步骤 0 → 1 → 2 → 3 → 4 → 5 → 6 → 7
刷新模式：  步骤 0 → R1 → R2 → R3 → R4 → R5 → R6
                              ↑ 共享步骤 3 的工具集（变量套用+mine+反面论据+透镜+二级效应）
```

### 0. 模式检测

1. 计算 `{slug}`（英文小写+短横线，中文保留）
2. 检查 `wiki/analysis/{slug}/` 是否存在
3. 存在且包含 report.md 或 follow-ups/ → 进入刷新模式（触发 CHECKPOINT: 模式确认）
4. 不存在 → 进入新课题模式（继续步骤 1）
5. 用户明确说"重新研究" → 强制新课题模式（备份旧目录）

## 刷新模式流程（R1-R6）

> **核心原则**：刷新模式不是"轻量维护"——深度必须等同新课题研究。已有 tracker 是起点不是约束。sweep 是定期体检，刷新模式是"重新做一遍深度研究，但站在已有积累上"。

### R1. 基线读取与 delta plan

**目的**：把已有研究读透，识别"上次判断 + 当前状态"的差距。

1. **读取已有内容**：
   - `report.md`（结论、持仓建议、核心发现）
   - `follow-ups/*.md`（特别是 macro-tracker 的变量地图、事件日历、跟踪记录、候选池）
   - `notes.md`（讨论记录、信息缺口）
   - `plan.md`（当时的研究问题、信息缺口）
   - 相关 `decisions/` 记录（当时决策依据）
   - `context/` 相关文件（个人上下文）

2. **对照当前时间，识别 delta**：
   - 上次报告结论时间 → 今天，过了多久？
   - 哪些核心数据已超过时效窗口？
   - 事件日历中哪些事件已过但结果未填？
   - tracker 中的"待验证变量"是否有结论？
   - 候选池（wiki-mine 挖出的盲区）状态如何？
   - 之前的"未解决问题"现在是否有答案？

3. **创建 `plan.md` 刷新记录**（追加，不覆盖原 plan）：
   ```markdown
   ## 刷新记录 {YYYY-MM-DD}

   **上次结论时间**：YYYY-MM-DD（距今 N 天/月）
   **上次核心判断**：{一句话}

   **本次刷新焦点**：
   - 数据更新：哪些核心数据过时了
   - 上次未解决问题：现在有答案了吗
   - 候选池变量：M1/M2/M3 是否升级或丢弃
   - 新盲区挖掘：再跑一遍 wiki-mine

   **刷新深度要求**：等同新课题研究（完整搜索+反面论据+透镜+二级效应）

   **进度**
   - [ ] R2 重新搜集资料
   - [ ] R3 深度分析（含盲区挖掘+反面论据+透镜+二级效应）
   - [ ] R4 delta report
   - [ ] R5 讨论确认
   - [ ] R6 入库 + 跟踪更新
   ```

🔴 **CHECKPOINT: 刷新范围确认**
向用户展示 delta plan 的核心内容，特别是：
- 上次结论时间 + 当前状态对比
- 本次刷新焦点
- 询问是否还有特别关注的方向

### R2. 重新搜集资料

**核心原则**：搜集成新课题一样彻底，不能因为"上次搜过"就跳过。

1. **基于 delta plan 重新搜索**：
   - 上次报告中所有过时数据，搜索最新值
   - 上次未解决问题，搜索是否有新答案
   - 事件日历已过事件的结果
   - 候选池变量的验证进展

2. **完整执行搜索策略**（同步骤 2 的分层搜索检查清单）：
   - 10 层搜索逐层覆盖
   - 中英文双语
   - 时效性规则（搜索词带当前年份）
   - 中文源 / SEC 文件 / 叙事变量扫描

3. **抓取新材料**：
   - 保存到 `materials/`，编号从上次最后一个继续（如已有 001-015，新材料从 016 起）
   - frontmatter 必须包含 `data_as_of`
   - 不重新抓取已抓取且未过时的材料

4. **多轮循环**（同步骤 2）：
   - 最多 3 轮搜索
   - 每轮汇报发现

### R3. 深度分析

**核心原则**：深度等同新课题研究。复用步骤 3 的全部工具集，但增加"对照上次"的特殊视角。

#### R3.1 变量模板套用 + wiki-mine 盲区挖掘（同步骤 3 变量部分）

1. 读全局变量库（registry / sectors / edges），特别是上次沉淀的变量
2. 套用板块模板
3. **调用 wiki-mine 再挖一轮盲区**——上次挖的盲区不等于这次的盲区，市场在变化
4. 对照上次 tracker 的候选池：
   - 上次的 M1-M8 等候选，现在信号是否强化？是否达到升级条件？
   - 是否有新的盲区出现？

#### R3.2 分析与对照（核心差异点）

逐篇阅读新材料（同步骤 3），但每条发现都要**对照上次报告**：

1. 创建/追加 `notes.md` 的 `## 刷新分析 {YYYY-MM-DD}` 节
2. **每条新发现必须对照**：
   - 上次报告有这个数据吗？变了多少？
   - 上次结论是否仍然成立？
   - 哪些假设链被打破？哪些被加固？
3. **失效判断处理**：
   - 上次结论中的某条失效 → 明确标注「❌ 上次判断失效：XXX（原因）」
   - 失效部分在 delta report 的「结论演变」节显式记录，不静默抛弃
4. **变量生命周期管理**：
   - 上次 tracker 中的变量，哪些已"消化完"可以关闭？
   - 哪些权重需要调整（★★☆ → ★★★ 或反向）？
   - 新挖出的盲区变量，按 wiki-mine 的候选池机制处理

#### R3.3 反面论据检查（同步骤 3 反面论据部分，强制执行）

特别注意：**上次的结论是否被新证据推翻？**这是刷新模式最关键的检查——人（和 AI）最容易陷入确认偏误，倾向维护原结论。

- 如果上次结论是"买入/加仓"，本次反方论点必须是"应该卖出/观望"
- 主动搜索反方证据（至少 3 个搜索词）
- **结论推翻规则**：如果新反方证据中有 2 条以上是上次报告无法解释的硬伤 → 必须下调评级，不能含糊带过

#### R3.4 透镜碰撞（同步骤 3 透镜部分，强制执行）

#### R3.5 二级效应扫描（同步骤 3 二级效应部分，强制执行）

特别注意：本次的新变量 + 上次遗留变量的**叠加效应**。

### R4. 输出 delta report + 更新 tracker

**两种产出，目的不同**：
- **delta report**：记录"上次到现在发生了什么变化、判断如何演变"
- **更新后的 tracker**：当前最新状态（macro-tracker 持续维护，追加而非覆盖）

#### R4.1 创建 delta report

文件：`wiki/analysis/{slug}/delta-{YYYYMMDD}.md`

```markdown
---
topic: 课题名称
type: delta-report
created: YYYY-MM-DD
previous_report: report.md (YYYY-MM-DD)
status: completed
---

## 结论演变

> **本次结论时间：YYYY-MM-DD** | 上次：YYYY-MM-DD（距今 N 天）

| 维度 | 上次判断 | 本次判断 | 变化原因 |
|------|---------|---------|---------|
| 操作方向 | 加仓 | 观望 | 核心假设X被新数据打破 |
| 核心驱动 | 变量A | 变量B | 主导变量切换 |
| 关键变量 | V101 = $X | V101 = $Y（+Z%） | 数据刷新 |

**结论变化一句话**：{本次相比上次的本质变化}

## 本次新发现

- 发现1（→ 来源：materials/016-xxx.md）
- 发现2（→ 来源：materials/017-xxx.md）

## 上次判断复盘

### 仍然成立
- {上次结论条目}：✅ 仍成立，原因/证据
- {上次结论条目}：✅ 仍成立

### 已经失效

## 新盲区发现

- M9（新挖）：变量描述
- M1（上次候选池）：升级到 registry，理由
- M3（上次候选池）：仍待验证，原因

## 持仓建议演变

<!-- 仅决策型研究 -->

| 维度 | 上次 | 本次 | 触发条件 |
|------|------|------|---------|
| 操作方向 | — | — | — |
| 止损线 | $XX | $YY | 触发原因 |
| 加仓信号 | — | — | — |

## 未解决问题

- 上次未解决 + 本次新出现

## 参考资料

- 本次新材料：materials/016 ~ 022
- 本次重新验证的上次材料：materials/003（重新确认）、materials/007（已过时）
```

#### R4.2 更新 tracker（macro-tracker）

**核心规则**：tracker 是"追加"而非"覆盖"。历史记录不动。

1. **关键变量与当前值**表：直接更新数值，但保留"上期"列做对比
2. **事件日历**：填入已过事件的实际值，添加新事件
3. **跟踪记录**：追加一条 `[auto: YYYY-MM-DD] 刷新模式更新` 记录，包含：
   - 本次刷新发现的核心变化
   - 哪些变量权重调整
   - 哪些候选池变量升级/丢弃
   - 失效的判断标注
4. **变量地图 / 对冲平衡表**：根据新数据更新（保留旧版本的截图或文字记录在跟踪记录里）

#### R4.3 更新 report.md（可选）

如果整体结论发生重大变化（如从"加仓"变"观望"），更新 `report.md` 的结论节：
- 保留旧结论（标注时间），追加新结论
- 不要覆盖旧判断——结论演变过程是重要资产

### R5. 讨论与补充

同步骤 5。讨论内容追加到 `notes.md` 的 `## 讨论记录 / ### YYYY-MM-DD：刷新讨论`。

### R6. 入库 + 跟踪更新

#### R6.1 知识入库（同步骤 6）

决策型 → delta report 保留在 analysis/，更新 decisions/（同 6b 决策联动）：
- 在已有决策记录的"复盘"节前添加 `## 决策更新 YYYY-MM-DD`
- 包含：新操作方向 + 变更理由 + 触发条件

知识型 → 关键新发现 → `wiki/sources/` 知识卡片

#### R6.2 变量沉淀（同步骤 7c）

特别关注：
- 候选池变量升级到 registry
- 已有变量的状态刷新
- 新传导关系加入 edges.md
- 失效变量标记关闭

#### R6.3 跟踪项更新

1. 更新 `follow-ups/*.md` 的 macro-tracker（已在 R4.2 完成）
2. 更新 `todos/active.md` 的跟踪项（cadence 触发）
3. 在 `wiki/log.md` 追加日志

#### R6.4 更新 changelog


## 新课题模式流程（步骤 1-7）

> 以下为完整流程，仅在新增课题时执行。刷新模式见上方 R1-R6。
> 前置检查（两种模式共用）：执行 `date`、检查 `WIKI_ROOT`、检查 `inbox/raw/wiki` 目录、检查 `wiki/analysis/`、列出 `context/`、检查 `decisions/`。

### 1. 开题

1. 读取 `wiki/index.md` 了解知识库全貌
2. 检查 `context/` 下是否有相关个人上下文（如涉及投资读 `finance.md`），纳入研究背景
3. 扫描 `wiki/sources/` 和 `wiki/topics/` 下已有相关内容，标记已知/未知
4. 创建 `wiki/analysis/{课题slug}/plan.md`（slug 规则：英文小写+短横线，中文保留）

plan.md 格式：
```markdown
---
topic: 课题名称
created: YYYY-MM-DD
status: active
type: book  <!-- 仅书籍课题添加此字段 -->
distill: true  <!-- 仅用户确认蒸馏时添加此字段 -->
---

## 研究问题
要回答的核心问题（1-3个）。

## 已有认知
来自知识库的已有内容（引用 sources/），如无则写"知识库暂无相关内容"。

## 信息缺口
还不知道、需要搜索的方向。

## 搜索计划
- 方向1：具体搜索关键词和目标
- 方向2：...

## 进度
- [x] 开题
- [ ] 搜集资料
- [ ] 研究分析
- [ ] 输出报告
- [ ] 知识入库
```

**书籍检测**：以下任一条件满足时，识别为书籍课题：
- 课题名称包含书名号《》
- 用户明确说这是一本书的研究（如"研究一下这本书"）
- **AI 判断课题名称可能是书名**：课题名称听起来像书名（如"XX研究"、"XX指南"、"XX与XX"、特定知名作品名），在开题阶段**主动询问**用户："这个课题是一本书吗？还是普通主题研究？"

识别为书籍后，在 plan.md 中添加 `type: book`，然后**依次处理**：

1. **书籍文件**：
   - 用户已提供路径 → 验证存在，记录到 plan.md
   - 用户要求下载 → 搜索书名 + PDF/电子书/下载，存入 `$WIKI_ROOT/data/{书名slug}.pdf`，记录路径
   - 用户未提及 → 询问：
     ```
     请提供书籍文件：
     - 书籍路径（如 data/xxx.pdf 或本地任意路径）
     - 如果没有文件，我可以尝试搜索下载 PDF
     ```

2. **蒸馏意愿**：使用 `cangjie-skill`（book2skill）的 RIA-TV++ 流程蒸馏方法论
   - 用户已明确要求蒸馏（如"提炼skill"、"蒸馏"、"输出skill"等表述）→ 直接设置 `distill: true`，告知用户："将使用 cangjie-skill 蒸馏方法论为可执行 skill，产出存放在 skills/ 目录并通过软链接接入全局 CC。"
   - 用户未提及 → 询问：
     ```
     是否同时蒸馏方法论为 skill？
     使用 cangjie-skill（RIA-TV++ 流程）从书中提取可执行方法论。
     - 是：研究 + 蒸馏一气呵成，产出知识层（report）+ 工具层（skills/）
     - 否：只做常规研究，产出知识层
     ```
   - 用户说否 → 不蒸馏

🔴 **CHECKPOINT: 蒸馏确认**
蒸馏是不可逆的高投入操作，必须满足以下全部条件才能执行：
- plan.md 已标记 `type: book`
- 用户已明确回复"是"（或等效表述）
- 用户未回复 → **停在步骤 1，不进入搜集资料**
- 用户回复"否" → `distill` 字段不设置，后续跳过步骤 8

**书籍文件处理**：

6. **调用 wiki-mine 挖掘盲区变量**（决策型研究强制执行）：已知变量套用完后，调用 `/wiki-mine` 从产业链漫游、社区异常词频、反向案例等机制发现 registry 里没有的盲区变量。挖掘产出的候选变量写入 tracker 的"待验证变量"区
7. **识别新变量**：研究过程中发现 registry 里没有的变量，标记为"🆕 待沉淀"（步骤 7c 处理）

**强制规则**：决策型研究产出的 macro-tracker 必须引用全局变量 ID（如"V101 尿素价格 $444"），不重复定义变量。

创建 `wiki/analysis/{课题slug}/notes.md`，逐篇阅读 materials/：

- 交叉验证不同来源
- 标记信息缺口
- **数据时效性检查**：对每个核心数据点，检查其 `data_as_of` 是否超过时效窗口（参见 CLAUDE.md）。过时数据标注 `⚠️ 数据截至 YYYY-MM-DD，距今 X 个月`，明确区分"回溯素材"和"当前状态"

- 最多 3 轮，避免无限搜索
- 每轮完成后向用户简要汇报发现

**跨课题关联发现**（分析完成后执行）：
1. 回顾核心发现，检查是否与 `wiki/analysis/` 下其他课题或 `wiki/topics/` 下其他专题有关联
2. 关联类型：因果链、共振、矛盾、对冲（参见 CLAUDE.md「跨课题关联」）
3. 发现关联时记录下来，在报告的「跨课题关联」节中输出，并创建关联分析卡片到 `wiki/connections/`

**反面论据检查**（决策型研究必须执行，知识型跳过）：

目的：对抗确认偏误——人类和 AI 都倾向于先有结论再找支持证据。这一步强制从"反方律师"视角审视结论。

1. **构造反方论点**：把当前结论反过来写。如结论是"推荐买入 X"，反方论点就是"X 现在不应买入"
2. **主动搜索反方证据**（至少 3 个搜索词）：
   - "[公司] bear case [YYYY]"、"[公司] overvalued overpriced"
   - "[公司] short thesis"、"why [公司] stock will fall"
   - "[公司] institutional selling"、"insider selling [公司]"
3. **列出看空/看多信号对比表**，写入 notes.md：

   | 维度 | 看多信号 | 看空信号 | 净方向 |
   |------|---------|---------|--------|
   | 基本面 | ... | ... | 多/空/平 |
   | 技术面 | ... | ... | |
   | 情绪/资金面 | ... | ... | |
   | 估值 | ... | ... | |

4. **结论合法性检查**：如果反方证据中有任何一条是**当前结论无法解释或绕不过的硬伤**，必须在报告中显式回应，不能含糊带过
5. **红灯触发**：如果净方向为"空"或反方有 2 条以上硬伤 → 评级从"买入"下调至"观望"，并在报告中明确写"等 X 信号确认后再评估"

**结论定价验证（MPV）**（决策型研究必须执行，知识型跳过）：

目的：对抗"好逻辑 ≠ 好交易"陷阱。逻辑成立只是必要非充分条件——市场可能已经 price in，导致结论正确但无赚钱空间（如电力设备层 GRID YTD +23%、推理层 Cerebras +94%，逻辑对但已部分定价）。与反面论据检查互补：**反方攻逻辑对不对，MPV 攻逻辑成立后还有没有交易价值**。

**三维定价评估**（每维用不同数据量化，禁止单看基本面数据就下"买"结论）：

| 维度 | 量化什么 | 数据源 | 判断标准 |
|------|---------|--------|---------|
| **① 逻辑成立度** | 基本面/产业证据 | 收入增速、backlog、交期、渗透率、份额 | 强/中/弱（≥1 可验证数据点）|
| **② 定价充分度** | 价格隐含预期 vs 实际 | forward PE/PS 反推隐含增速、YTD/从低点涨幅、距 52 周高位置、PEG | 未定价 / 部分 / 充分 |
| **③ 拥挤度** | 资金与共识 | ETF 资金流（FactSet/ETF.com）、13F 机构持仓集中度、空头权益、卖方目标价上调密度、Reddit/媒体渗透率 | 空缺 / 合理 / 拥挤 |

**定价矩阵 → 操作含义**（核心产出，每个"买入/加仓/埋伏"结论必须给出定位）：

| 逻辑 | 定价 | 拥挤 | 判定 | 操作 |
|------|------|------|------|------|
| 强 | 未定价 | 空缺 | 🟢 真机会 | 埋伏 |
| 强 | 部分 | 合理 | 🟡 顺势 | 小仓位 / 等回调 |
| 强 | 充分 | 拥挤 | 🔴 负偏 | 不追，等 shakeout |
| 强 | 充分 | 空缺 | 🟠 异常 | 调研为何无共识（陷阱 or 机会）|
| 弱 | — | — | ⚫ 放弃 | — |

**执行要求**：
1. 每个"买入/加仓/埋伏"结论必须填三维评估表 + 定价矩阵定位，缺一不可
2. 关键数据回原始源（forward PE→公司 IR/SEC、ETF 资金流→FactSet/ETF.com、空头→exchange/alphavantage）
3. **🔴 红灯规则**：结论落在"强逻辑 + 充分定价 + 拥挤"（负偏格）→ 操作从"加仓"下调至"观望/等回调"，报告必须写"逻辑对但已定价，等 X 回调"，**禁止给"逻辑好所以买"的建议**
4. 超越纯量化的补充工具（可选但推荐）：**历史定价锚**（上轮同类主线见顶估值，如 2000 Cisco PE 峰值、2021 SaaS EV/Rev 峰值）、**共识渗透率温度计**（卖方看多报告占比/媒体标题情绪/Reddit 讨论量）、**反身性钟摆位置**（Marks：恐慌/共识/狂热）
5. 产出落盘：MPV 三维表 + 定价矩阵追加到 `notes.md` 的 `## 结论定价验证` 节；`report.md` 的"持仓建议"必须引用定价矩阵定位（如"🟡 顺势，等回调 10-15%"）

**与反面论据检查的关系**：反方检查（上一节）攻"逻辑是否成立"，MPV 攻"逻辑成立后是否还有交易价值"。**两者都过的结论才进"买入"**——只过反方没过 MPV = 逻辑对但已定价，只能"观望/等回调"。

**透镜碰撞**（决策型研究必须执行，知识型研究可选）：

目的：让蒸馏出的方法论主动审视研究结论，产出单看任何一篇材料都得不出跨界洞察。AI 不会自发创造关联，必须用方法论作为强制视角。

1. **触发事件构造**：把研究的核心结论 + 关键变量提炼为"事件描述"（1-2 句话）
2. **调用 wiki-lens**：把事件描述作为输入，执行 wiki-lens skill 的完整流程
   - wiki-lens 自动从 `wiki/sources/` 中带「作为透镜如何使用」字段的卡片里选 top 3-5 个透镜（强制至少 1 个跨界）
   - 每个透镜按其「强制问题」产出一条判断
   - 透镜之间碰撞出新洞察
3. **产出落盘**：
   - 透镜判断概要追加到 `notes.md` 的 `## 透镜碰撞` 节
   - 非显然洞察独立写入 `wiki/connections/YYYY-MM-DD-{课题}-lens.md`（文件名加 `-lens` 后缀以区分）
   - 在 report.md 的「## 跨课题关联」节末尾引用："透镜碰撞产出 N 条洞察，详见 [connections 文件名]"
4. **质量门槛**：
   - 透镜判断必须有方向性（涨/跌/持续/反转），不能是"需关注"
   - 新洞察必须有反直觉成分，否则不写入 connections
   - 至少 1 个透镜来自研究课题所属领域之外（强制跨界）

**二级效应扫描**（决策型研究必须执行，知识型跳过）：

目的：发现"非显而易见的跨域影响因子"——即那些不在直接因果链上、但通过间接路径影响目标标的的事件和变量。这类因子人类凭直觉容易忽略（如大型 IPO 冻结资金导致科技股被砸），AI 受限于训练数据也难以自主生成，必须通过结构化流程强制扫描。

1. **收集近期事件清单**：从已有材料和搜索结果中，列出所有近期已发生或即将发生的重要事件（不限领域——IPO、政策、地缘、大型交易、技术发布等）
2. **逐事件推演间接影响路径**：对每个事件问——"这个事件通过什么间接路径可能影响目标标的？"强制跨越以下 8 个域思考：
   - 资本流动（资金冻结/释放、ETF申赎、期权到期、基金调仓）
   - 产业链（上游供应变化传导到下游需求）
   - 情绪/叙事（一个板块的叙事溢出到另一个板块）
   - 政策/监管（A 国政策引发 B 国连锁反应）
   - 跨资产（债券收益率→估值、汇率→出口、商品→成本）
   - 竞品/替代（A 公司的利好是否在抢 B 公司的资金/注意力）
   - 日历效应（季节性、周期性事件的时间窗口）
   - 历史类比（以前有没有类似事件？结果怎样？）
3. **搜索历史类比**：对每个识别出的间接影响路径，搜索"以前是否发生过类似事件→结果如何"。用搜索词如："[事件类型] impact on [资产类别] historical"、"historical IPO capital drain effect tech stocks"
4. **输出非显而易见的影响因子**：在 notes.md 中记录找到的间接影响路径和历史类比证据，标注可信度。在报告的「详细分析」中以 `### 二级效应：XXX` 子节输出

**用户反馈点**：
- "继续深入 XX" → 补充搜索
- "差不多了，出报告" → 进入步骤 4
- 不回复 → 继续下一轮

### 6. 知识入库 & 决策联动

🔴 **CHECKPOINT: 入库确认**
入库是不可逆操作（知识库是长期积累的资产），必须向用户确认入库方案后执行。
- 向用户说明产出类型判断（知识型/决策型）及对应的入库操作
- 用户确认 → 执行入库
- 用户调整 → 按用户要求修改后重新确认
- 未获确认 → 停在步骤 6，不入库

#### 6a. 判断产出类型

**知识型**（方法论、原理、通用框架）：
1. 创建知识卡片到 `wiki/sources/`，`source: research`，`origin: analysis/{slug}/report.md`
2. 更新相关 topic，达到 3 篇时创建新 topic
3. 更新 `wiki/index.md` 和 `wiki/log.md`

**决策型**（时效性分析、个股研究、个人建议）：
- 报告保留在 `analysis/`，不入库
- 写入 `wiki/log.md`
- **必须执行 6b 决策联动**

**判断规则**：
- 特定个股/行业当前状态 → 决策型
- 通用方法论/原理/框架 → 知识型
- 不确定时问用户："这是时效性分析，建议保留在 analysis/ 不入库，可以吗？"

#### 6b. 决策联动（决策型研究强制执行）

当研究涉及个人持仓/操作建议时（即 report 包含"持仓建议"节），**必须**在 `decisions/` 下创建或更新决策记录。

**新建决策记录**（无同名标的 open 决策时）：

文件名：`YYYY-MM-DD-{标的简称}持仓决策.md`

```markdown
---
title: {标的}持仓决策
date: YYYY-MM-DD
tags: [投资]
status: open
review_date: YYYY-MM-DD（默认 3 个月后）
context: analysis/{课题名}
---

## 决策
{从 report 的"持仓建议"节提取操作方向和止损线}

## 当时理由
{从 report 的"核心发现"节提取 2-3 条关键依据}

## 预期
{从 report 的"持仓建议"节的情景分析提取}

## 复盘
（待复盘）
```

**更新已有决策**（decisions/ 下已有同一标的的 open 状态决策时）：
- 不创建新文件，而是在"复盘"节前添加 `## 决策更新 YYYY-MM-DD`
- 包含：新操作方向 + 变更理由（什么变了导致决策改变）
- 更新 `review_date` 为新日期 + 3 个月

创建/更新后向用户确认决策记录内容。

### 7. 跟进（强制，不可跳过）

研究完成后 **必须** 执行：

**7a. 变量梳理（决策型研究必须，知识型跳过）**

从研究材料中系统提取影响结论的所有变量，梳理传导关系：

1. **提取变量**：回顾 materials/ 和 notes.md，列出所有影响结论的变量（基本面、技术面、消息面、宏观、政治、情绪）
2. **画传导关系**：用箭头连接变量间的因果关系（谁影响谁、谁被谁影响）
3. **分层标注**：将变量分为"当前走势驱动"（1-4周内主导股价的变量）和"长期结构关键"（季度-年级别的核心变量），标注影响权重和转折信号
4. **标出关键事件**：哪些事件会改变哪些变量（财报日期、CPI发布、FOMC、产品发布等）
5. **定义确认规则**：几个信号同时出现才确认趋势/触发操作（避免单次数据就做判断）

将以上内容整合为一个 `macro-tracker` 类型的 follow-up（格式见 CLAUDE.md「决策型跟踪项」）。

**7b. 创建跟踪项**

1. 回顾 report.md 的核心发现和未解决问题
2. 识别需要持续跟踪的项（数据变化、业务指标、政策动态、技术演进）
3. 向用户确认：
   ```
   以下发现可能需要持续跟进：
   1. xxx（如：PDD 半托管模式占比变化）
   2. xxx

   同时，我梳理了影响结论的关键变量和传导关系，建议创建一个变量跟踪项（macro-tracker）。
   需要创建跟踪项吗？可以选择全部或指定编号。
   ```
4. 用户确认后：
   - 在 `follow-ups/` 下创建跟踪文件（格式见 CLAUDE.md）
   - 决策型课题：创建1个 macro-tracker（变量地图+事件日历+确认规则），替代散装的多个独立 follow-up
   - 在 `todos/active.md` 对应分类的 `### 长期` 下添加：`- [ ] 跟进：{课题} - {项} \`cadence: monthly\` \`created: YYYY-MM-DD\``
5. 用户说不需要 → 仅记录到 log.md

### 7c. 变量沉淀（决策型研究必须执行）

目的：把本次研究发现的变量和传导关系沉淀到全局变量库，让下次分析其他标的时可以复用。这是变量层"自动维护"的核心机制——**研究完成必须沉淀，避免经验流失**。

1. **更新 registry.md**：
   - 把 notes.md 中标记为"🆕 待沉淀"的新变量加入 `wiki/variables/registry.md`，分配新 ID（按板块 ID 段）
   - 已存在变量的状态有更新 → 刷新 registry 中的"当前状态"列和"截至"日期
   - 在 registry 末尾追加维护记录（如"- 2026-07-05：从 XXX 研究沉淀 V101-V110"）

2. **更新 edges.md**：
   - 本次验证了某条传导关系（如"霍尔木兹复航 → 尿素 -52%"）→ 加入 edges.md，标注 ✅ 已验证
   - 本次发现新的传导关系（如"飓风 → Donaldsonville 停产 → 全球供给 -8%"）→ 加入 edges.md
   - 已有边的验证状态有变化 → 更新（如从 ⏳ 待验证 → ✅ 已验证）

3. **创建/更新板块模板**（如适用）：
   - 如果是新板块 → 创建 `wiki/variables/sectors/{板块}.md`（套用 CLAUDE.md 中的格式）
   - 已有板块模板 → 补充本次发现的板块特性、已知陷阱、标的差异化要点

4. **创建关键变量档案**（仅反复出现、影响多个标的的变量）：
   - 某个变量在 registry 里被 ≥3 个标的引用，或影响 ≥2 个板块 → 创建 `wiki/variables/variables/{slug}.md` 深度档案
   - 套用 CLAUDE.md 中的变量档案格式（含历史波动、跟踪方式、关联课题）

5. **macro-tracker 引用变量 ID**：
   - 检查本次产出的 macro-tracker 是否引用了全局变量 ID
   - 没有引用的 → 补充引用（如"V101 尿素价格 $444"而不是孤立写"尿素价格 $444"）

**质量门槛**：
- 每条新加入的 edge 必须有"机制"说明（不能只写"A → B"）
- 每条新加入的 variable 必须有"传导下游"字段
- 板块模板的"已知陷阱"必须有具体来源（哪次研究、什么教训）

向用户简要汇报沉淀结果："本次研究沉淀了 N 个新变量、M 条传导关系到全局变量库。"

### 8. 蒸馏（仅书籍课题 + 用户确认时执行）

当 plan.md 同时标记 `type: book` 和 `distill: true` 时，在研究完成（步骤 4-7）后执行蒸馏。

#### 8a. 调用 cangjie-skill

使用 `cangjie-skill`（book2skill）的流程，从研究材料中蒸馏可执行方法论：

1. 将 materials/ 和 notes.md 作为输入，调用 cangjie-skill 的提取流程
2. 输出目录设为 `wiki/analysis/{课题slug}/skills/`（而非 cangjie-skill 默认的 `books/` 目录）
3. cangjie-skill 产出：
   - `skills/INDEX.md` — 候选 skill 索引
   - `skills/candidates/` — 待筛选的 skill 候选
   - `skills/rejected/` — 不达标的候选
   - `skills/{skill名}/SKILL.md` — 最终产出的 skill
   - `skills/{skill名}/test-prompts.json` — 测试提示词

#### 8b. 创建全局软链接

蒸馏完成后，将产出的 skill 通过软链接接入全局 CC：

1. 确保 `~/.claude/skills/books/` 目录存在
2. 为每个产出的 skill 创建软链接：
   ```bash
   ln -sf "$WIKI_ROOT/wiki/analysis/{课题slug}/skills/{skill名}/SKILL.md" \
          ~/.claude/skills/books/{课题名}-{skill名}.md
   ```
3. 更新 `~/.claude/skills/books/_index.md`，添加新条目：
   ```markdown
   - [{课题名} - {skill名}](books/{课题名}-{skill名}.md) — 一句话摘要
   ```

4. 向用户汇报蒸馏结果："从《{书名}》中蒸馏出 N 个方法论 skill：{skill名列表}，已链接到全局 CC 可复用。"

## 失败模式编码

以下失败场景按 if-then 三段式处理：**如果**出现该情况 → **则**执行指定动作 → **否则**继续正常流程。

### FM-1: 搜索无结果

**如果**某个搜索方向的 3 轮查询均返回零相关结果（或全部是已抓取的重复内容）：
**则**：
1. 在 notes.md 记录该方向搜索失败，附上已尝试的关键词
2. 向用户汇报："搜索方向「{方向名}」未找到有效结果，已尝试关键词：{列表}。建议：{调整关键词/换搜索源/跳过该方向}。"
3. 等待用户决定（调整关键词 / 跳过该方向 / 终止研究）
**否则**：正常抓取和保存。

### FM-2: 网页抓取失败

**如果** `mcp__web_reader__webReader` 返回错误（超时、403、付费墙、页面无法解析）：
**则**：
1. 跳过该 URL，不重试（避免浪费时间）
2. 在 notes.md 的 `## 抓取失败记录` 节追加：`- {URL} | 失败原因: {错误类型} | 日期: YYYY-MM-DD`
3. 如果该来源是搜索计划中某方向的唯一结果，回到搜索步骤补充其他来源
**否则**：正常保存到 materials/。

### FM-3: PDF 下载失败

**如果**书籍课题需要下载 PDF，但搜索后未找到可下载链接，或下载的文件损坏/加密/非 PDF 格式：
**则**：
1. 告知用户："未找到《{书名}》的可下载 PDF，已尝试的来源：{列表}。"
2. 提供选项让用户选择：
   - 用户提供文件路径
   - 用户明确说"不用原文，找其他资料也行" → 降级为普通主题研究（移除 `type: book`）
   - 终止研究
3. **禁止**自行决定用书评、摘要、总结文章替代原文
**否则**：正常保存文件并记录路径。

### FM-4: 信息严重不足

**如果**经过 3 轮搜索后，materials/ 中有效材料少于 3 篇，且 notes.md 中未解决问题多于已解决问题：
**则**：
1. 汇总已获取信息和剩余缺口，向用户诚实汇报：
   ```
   信息不足，当前情况：
   - 已获取：{简要列出已有材料}
   - 缺口：{列出关键未解答问题}
   - 建议：{扩展搜索方向 / 降低研究范围 / 终止研究}
   ```
2. 由用户决定是否继续（扩展搜索 / 缩小范围 / 终止）
3. 用户选择终止 → 产出"研究中断"报告（status: interrupted，记录已获取的信息和中断原因）
**否则**：正常进入分析阶段。

### FM-5: 课题目录已存在

**如果** `wiki/analysis/{课题slug}/` 目录已存在：
**则**：
1. 触发步骤 0 模式检测，进入「## 两种执行模式」的 CHECKPOINT: 模式确认
2. 用户选择刷新模式 → 走 R1-R6 流程
3. 用户选择重新研究 → 备份旧目录为 `{slug}-v1`，走新课题模式
4. 用户取消 → 停止
**否则**：正常创建课题目录，进入新课题模式。

### FM-6: 知识库目录缺失

**如果**前置检查发现 `$WIKI_ROOT/wiki/` 或其子目录不存在：
**则**：
1. 输出明确的错误信息："知识库目录缺失：{路径} 不存在。WIKI_ROOT={当前值}。请检查：1) WIKI_ROOT 是否指向正确的 ai-wiki 路径；2) 是否已运行知识库初始化。"
2. 停止执行，等待用户修复
**否则**：正常继续。

## 反例与黑名单

以下是 wiki-research **明确禁止**的行为。违反任何一条即视为流程违规。

### 反例 1: 用搜索摘要替代书籍原文

**禁止**：书籍课题找不到 PDF 时，用搜索引擎返回的书评、读书笔记、豆瓣短评、公众号摘要、知乎回答来"拼凑"书籍内容。
**原因**：二手内容经过他人筛选和解读，遗漏关键论证链条，无法还原作者完整方法论。
**例外**：用户明确说"不用原文，找其他资料也行"。

### 反例 2: 未经用户确认就自动蒸馏

**禁止**：识别为书籍课题后，不询问蒸馏意愿就直接调用 cangjie-skill 执行蒸馏。
**原因**：蒸馏是高耗时操作（通常需要多轮交互），且会修改全局 skill 目录和软链接。未经确认的蒸馏浪费 token 并污染 skill 库。
**正确做法**：必须通过 🔴 CHECKPOINT: 蒸馏确认，等待用户明确回复。

### 反例 3: 开题后直接跳到搜索

**禁止**：创建 plan.md 后不展示给用户确认，直接开始搜集资料。
**原因**：研究方向可能有偏差（问题定义不准、漏掉关键维度、已有知识覆盖等），用户确认是纠偏的关键机会。
**正确做法**：必须通过 🔴 CHECKPOINT: 开题确认，等待用户回复后才能进入步骤 2。

### 反例 4: 时效性数据不标注截至日期

**禁止**：在报告和材料中引用量化数据时不标注"截至 YYYY-MM-DD"。
**原因**：研究报告的生命周期可能跨越数月甚至数年，无时效标注的数据会被误认为当前值，导致错误决策。
**正确做法**：所有量化数据标注 `[截至 YYYY-MM-DD]`，frontmatter 写 `data_as_of`。

### 反例 5: 信息冲突时自行"裁决"

**禁止**：不同来源数据矛盾时，AI 自行判断哪个"更可信"并只保留一个。
**原因**：矛盾本身是重要信号（可能反映数据源差异、时间差异、立场差异），用户需要看到完整画面才能判断。
**正确做法**：标注 `[冲突]`，记录各方来源和依据，向用户汇报。

### 反例 6: 决策型研究跳过决策联动

**禁止**：报告包含"持仓建议"节但不在 `decisions/` 下创建或更新决策记录。
**原因**：决策记录是后续复盘（wiki-review）的输入，缺失则无法验证决策质量和改进决策流程。
**正确做法**：步骤 6b 是决策型研究的强制步骤，不可跳过。

### 反例 7: 只搜索英文源或只搜索中文源

**禁止**：整个研究过程只使用一种语言的搜索源。
**原因**：中英文信息生态差异大，单一语言会系统性遗漏关键信息（如 A 股的中文研报、美股的 SEC 文件）。
**正确做法**：同一主题至少中英文各搜一次，搜索策略中有明确规定。

### 反例 8: 擅自修改报告模板结构

**禁止**：自由增删改 report.md 的章节（如加"多空辩论"节、删"未解决问题"节、把"结论"拆成多个子节）。
**原因**：报告模板是统一格式，保证所有研究产出可比、可索引、可复盘。自由发挥导致后续 wiki-sweep、wiki-review 等流程无法正确解析。
**正确做法**：严格遵循步骤 4 的模板，补充内容放在"详细分析"的子问题分节中。
