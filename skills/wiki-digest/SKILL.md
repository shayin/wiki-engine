# Digest — 文章处理流程

## 触发条件

- 用户发送 URL（http/https 开头）
- 用户说"处理"、"收录"、"digest" + 附带链接或文本
- 用户发送一段长文本（非 URL，视为自己写的文章，`source: self`）

**缺少输入时**：回复"请提供文章链接或直接粘贴文章内容。"并停止执行。

**批量 URL**：用户一次发送多个 URL 时，按顺序逐个处理，每个走完整流程。每篇完成后简短回复，全部完成后汇总。

## 前置检查

1. 检查环境变量 `WIKI_ROOT` 是否设置。未设置则报错："未设置 WIKI_ROOT 环境变量，无法定位知识库。请运行 `echo $WIKI_ROOT` 检查。"
2. 检查 `$WIKI_ROOT/inbox/`、`$WIKI_ROOT/raw/`、`$WIKI_ROOT/wiki/` 三个目录是否存在
3. 检查 `$WIKI_ROOT/wiki/index.md` 和 `$WIKI_ROOT/wiki/log.md` 是否存在
4. 不满足时报错并提示检查 WIKI_ROOT 配置

## 反例黑名单

以下行为**严格禁止**，任何场景下都不得违反：

1. **不要自动删除 inbox 文件** — 只能移动到 `raw/`，绝不能 `rm`
2. **不要修改已有 sources 卡片内容** — 如需补充，只追加「更新记录」子段，不改动原有正文
3. **不要为单篇文章创建 topic** — 至少 2 篇有实质关联才创建主题页面
4. **不要跳过重复检查直接写入** — 每篇文章必须执行步骤 2 的重复检测
5. **不要在 index.md 中使用相对路径** — 所有链接使用 `$WIKI_ROOT` 起始的绝对路径或 `sources/YYYY-MM-DD-xxx.md` 标准路径
6. **不要忽略 webReader 抓取失败** — 抓取失败必须记录到 `log.md`，不能静默跳过
7. **不要创建标签数量超过 5 个的卡片** — 标签上限 5 个，超出时优先保留最相关的
8. **不要把 AI 生成摘要当作原文归档** — `raw/` 中只能保存原始抓取内容，摘要仅存在于 `sources/` 卡片

## 执行步骤

### 1. 获取内容

**URL 模式**：
- 使用 `mcp__web-reader__webReader` 抓取，参数：`retain_images: true, return_format: markdown`
- 失败时回退 `mcp__web_reader__webReader`
- 都失败则告知用户，建议手动粘贴正文

**微信公众号特殊处理**：
- `mp.weixin.qq.com` 链接 >50% 概率抓取失败或内容不全（特征：返回空内容/只有标题/<500字）
- 如果是微信链接，优先提示用户："微信文章可能抓取不完整，如果内容不对请直接粘贴正文。"
- 抓取后检查内容长度，<500 字则建议用户手动粘贴

**长文本模式**：
- 直接使用用户文本，标记 `source: self`

**超长内容处理**（>10000 字）：
- 知识卡片的「关键要点」提炼 5-8 条（而非 3-5 条）
- 原文照常归档到 raw/，不截断

**提取用户备注**：
- 用户消息中除 URL 外的额外文字（如"这篇很重要"、"关于XX的"）提取为备注
- 备注用于辅助打标签和生成摘要

### 2. 疑似重复检查

检查 `wiki/sources/` 是否已有：
- 相同 URL（匹配 frontmatter `origin` 字段）
- 高度相似标题（模糊匹配）

如命中，暂停执行，输出 CHECKPOINT：疑似重复，提示用户："这篇文章可能已收录为「{已有标题}」，确认是否重复？"确认重复则停止，用户说不是重复则继续。

### 3. 存入 inbox/

保存到 `inbox/YYYY-MM-DD-简短标识.md`：
- 简短标识：标题前 2-4 个关键词，短横线连接，全小写，去特殊字符
- 示例：`2026-05-14-rust-async-runtime.md`

```markdown
---
url: 原始链接（如有）
fetched: YYYY-MM-DD HH:mm
status: pending
---

原始文章标题

原始正文内容...
```

### 4. 生成知识卡片

先读 `wiki/index.md` 了解知识库全貌，再按 CLAUDE.md 中「知识卡片格式」生成：

**source: article（外部文章）**：
- frontmatter：title、date、tags、source、origin、related
- 一句话摘要（50 字内）
- 关键要点（3-5 条）
- 原文链接

**source: self（用户写的）**：
- 保留原文不重写
- 只做分类、打标签、建关联
- 摘要从原文首段提取或让用户补充

**标签规则**：
- 两层：领域（技术/股票/生活/工作/读书...）+ 具体主题（Rust/量化交易...）
- 优先复用 `wiki/sources/` 下已有标签
- 每篇 2-5 个标签
- related 通过搜索 wiki/ 找相关页面，用 `[[]]` 双链

**关联发现**（在生成卡片时同步执行）：
1. 读 `wiki/index.md` 获取已有文章列表
2. 基于新文章的标签，搜索 `wiki/sources/` 和 `wiki/topics/` 下的相关文件
3. 精读匹配到的文件标题和摘要，判断是否存在语义关联（不仅仅是同标签）
4. 将有关联的文章/主题加入卡片的 `related` 字段
5. 如果发现跨领域的隐含关联（如一篇讲技术趋势，一篇讲投资标的），优先记录

关联类型优先级：
- 直接相关（同一公司/同一技术/同一事件）→ 必须关联
- 间接影响（政策→行业→公司链路）→ 应该关联
- 方法论可迁移（A 领域的方法适用于 B 领域）→ 建议关联（标注迁移路径）
- 纯粹标签相同但内容无关 → 不关联

### 5. 写入 wiki/sources/

保存到 `wiki/sources/YYYY-MM-DD-简短标识.md`

### 6. 归档原文

创建 `raw/YYYY-MM/`（如不存在），将 inbox/ 中的原文移过去。

### 7. 更新 wiki/index.md

在「最近更新」追加：
```
- YYYY-MM-DD: [文章标题](sources/YYYY-MM-DD-简短标识.md) #标签1 #标签2
```
更新统计数字。

**写入安全策略**：
- 写入前先读取当前 index.md 内容，校验文件结构完整性（frontmatter + 最近更新 section 存在）
- 校验失败则中止写入，告知用户"index.md 结构异常，请手动检查"
- 批量处理时累积所有变更，最后一次写入（避免多次读写同一文件）

### 8. 检查 topic 更新

检查本次文章的标签：
- 如果 `wiki/topics/` 下已有对应主题页面 → 追加到关联文章列表
- 如果与已有 sources 有**实质性关联**（见下方硬标准）但无 topic → 创建轻量主题页面（哪怕只有 2 篇）
- 标签重叠但内容无实质关联 → 不创建

**实质性关联三项硬标准**（满足任一即可创建 topic）：
1. **同一实体**：两篇文章讨论同一公司/技术/事件/人物
2. **结论影响前提**：A 文章的结论直接影响 B 文章的前提假设
3. **合答大问题**：两篇文章从不同角度回答同一个宏观问题

### 9. 检查跟进项关联

扫描 `wiki/analysis/*/follow-ups/` 下 `status: active` 的跟进项：
- 文章标签与跟进项 topic 匹配，或文章内容包含跟进项跟踪的指标/数据
- 匹配到时暂停执行，输出 CHECKPOINT：跟进项确认，提醒用户："这篇文章跟你跟进的「{跟进项名}」相关，要更新跟踪记录吗？"
- 用户确认后追加到跟进项的 `## 跟踪记录`：`- **YYYY-MM-DD**: 更新——xxx，来源 [[sources/本文]]`

### 10. 跨课题关联检查（轻量）

检查新文章是否与知识库中**其他课题**有跨专题关联：

1. 读取 `wiki/connections/index.md` 了解已有关联
2. 基于新文章内容，检查是否与 `wiki/analysis/` 下已有课题或 `wiki/topics/` 下其他专题存在语义关联
3. 关联类型判断：因果链（A→B）、共振（同步变化）、矛盾（结论冲突）、对冲（A利好=B利空）
4. **发现关联时**：创建关联分析卡片到 `wiki/connections/YYYY-MM-DD-简短标题.md`，更新 `wiki/connections/index.md`
5. **无关联时**：跳过，零额外成本

注意：仅基于标题+标签+摘要做一级判断，不做全文语义分析。关联卡片的「具体分析」可以简短。深度关联留给研究过程和 sweep。

**批量性能策略**：
- 处理 N 篇文章时，前 N-1 篇跳过步骤 10 的跨课题关联扫描
- 最后一篇处理完毕后，统一对所有已处理文章执行一次跨课题关联扫描
- 避免每篇文章都读取 `wiki/connections/index.md` 和 `wiki/analysis/`，减少 IO 开销

追加：`- YYYY-MM-DD HH:mm: 处理了「文章标题」→ sources/YYYY-MM-DD-简短标识.md`

### 11. 回复用户

```
已收录「文章标题」

一句话摘要。

#标签1 #标签2
```

## 错误处理

| 场景 | 处理 |
|------|------|
| 缺少输入 | 提示发送链接或粘贴正文 |
| 抓取失败 | 告知用户，建议手动粘贴 |
| 内容 <100 字 | 告知用户，确认是否仍要收录 |
| 文件名冲突 | 追加序号（`-2`） |
| 疑似重复 | 提示用户确认 |
| index.md 写入冲突 | 中止写入，告知用户"index.md 结构异常，请手动检查" |
| 步骤 8 判定边界 | 标签重叠但无实质性关联 → 不创建 topic；不确定时记录到 log.md 等用户确认 |
| 步骤 9 跟进项匹配 | 无精确匹配时不强行关联；标签部分匹配时提示用户但默认不关联 |

## 通知输出

cron 自动处理时，回复末尾**必须**输出一行 NOTIFY（供 cron wrapper 解析）：

```
NOTIFY: digest 完成：处理了 N 篇文章（{标题列表}）
```

- 有文章：`NOTIFY: digest 完成：处理了 3 篇文章（Rust异步、量化策略、新能源）`
- inbox 空：不输出（由 wrapper 的 ALL CLEAR 分支处理）

交互模式（用户手动触发）也需要输出这行，不影响用户阅读。

## 关联资源

- 知识卡片格式：参见 `CLAUDE.md` 中「知识卡片格式」
- 主题页面格式：参见 `CLAUDE.md` 中「主题页面格式」
- 抓取工具：`mcp__web-reader__webReader`（主）、`mcp__web_reader__webReader`（备）
- cron 调度：`.cron/config.sh` 中的 DIGEST_TIME 配置

## 路径约定

所有路径基于 `$WIKI_ROOT`。操作前先验证 `WIKI_ROOT` 环境变量已设置且 `$WIKI_ROOT/inbox/`、`$WIKI_ROOT/raw/`、`$WIKI_ROOT/wiki/` 目录存在。
