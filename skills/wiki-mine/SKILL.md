# wiki-mine — 变量挖掘

## 核心理念

**搜索是 top-down（演绎）**：你先猜到关联（"尿素价格影响 CF"），再去搜——只能验证已知假设。

**挖掘是 bottom-up（归纳）**：让 AI 漫游大量信息，让"信号"自己浮现，再反向回带到标的——发现未知变量。

传统关键字搜索的盲区：必须先有关联才能搜。本 skill 的核心是**不带关键字的漫游**——先浏览产业链、社区、新闻、研报，让反复出现的"信号"告诉你什么重要，而不是 AI 凭训练数据猜。

**散户视角**：挖掘的目标不是比机构先知道（信息差打不过），而是：
1. 发现机构忽略的维度（冷门变量、第二三跳传导链）
2. 发现市场错误定价（结构性被低估/高估）
3. 发现被忽略的风险因子（避免踩坑）
4. 发现长期催化剂（事件性变量提前布局）

**找盲区，不追热点。**

## 触发条件

- 用户显式调用："挖变量"、"找盲区"、"mine 一下 XX"、"变量体检"
- wiki-research 步骤 3「变量模板套用」前自动调用（新课题）
- wiki-sweep 维度 M 定期体检时调用（已有标的）

**缺少标的代码时**：回复"挖哪个标的的变量？"并停止执行。

## 前置检查

1. **获取准确时间**（必须）：执行 `date "+%Y-%m-%d %H:%M %a"` 作为时间锚点
2. 检查 `WIKI_ROOT` 环境变量
3. 读取 `$WIKI_ROOT/wiki/variables/registry.md` 获取已知变量清单
4. 读取 `$WIKI_ROOT/wiki/variables/edges.md` 获取已知传导关系
5. 如果标的有 macro-tracker（`$WIKI_ROOT/wiki/analysis/{标的}/follow-ups/*macro-tracker*.md` 或 `*关键变量总览*.md`），读取已有变量
6. 判断标的所属板块，读取对应的 `$WIKI_ROOT/wiki/variables/sectors/{板块}.md`（如果存在）

## 6 种挖掘机制

按"发现盲区"价值排序：

### 机制 A（P0）：产业链漫游（BFS）

**核心**：不搜标的本身，沿产业链走。第二跳、第三跳才是新变量的来源。

**方法**：
1. 从标的基本面出发，确定 3-5 个"锚点"（主营业务、核心产品、关键工厂、主要客户）
2. 对每个锚点，列出产业链节点：
   - **上游**：原材料、能源、设备、技术、运输
   - **下游**：客户、应用、替代品
   - **平行**：竞争对手、替代技术
3. 每走到一个节点，停留问："这里有什么变量会传导到标的？"
4. 对每个候选变量，估算传导路径和量化系数

**关键**：不要直接搜公司名。搜"行业/产品/技术 + 趋势/瓶颈/短缺/政策"。

**跳数规则**（详见下方"停止规则"）：
- 软警告线：5 跳（每多 1 跳必须说明为什么更可能找到盲区）
- 硬上限：10 跳（强制终止，但记录 pending 线索，最后问用户）

**示例**（CF Industries）：
```
第1跳（直接，已知为主）：
  尿素价格（V101）—— 已有
  Henry Hub 天然气（V102）—— 已有

第2跳（产业链中间节点，主战场）：
  天然气管道运力 → 传导可量化 → 候选
  美国铁路运力（BNSF/UP 罢工）→ 传导可量化 → 候选
  密西西比河水位（驳船运输）→ 历史上 2022 冲击过化肥 → 候选

第3跳（远端）：
  转基因固氮作物 → 长期需求 → 弱候选
  巴西大选 → 巴西进口政策 → 传导断链风险高 → 停止
```

### 机制 B（P0）：社区异常词频扫描

**核心**：让真实用户讨论告诉你什么重要，不靠 AI 凭训练数据猜。

**方法**：
1. 识别相关社区：标的对应的 subreddit、雪球、Seeking Alpha、行业论坛（如化肥→r/agriculture，半导体→r/hardware）
2. 抓取最近 30-90 天热帖/高赞评论
3. 做**词频分析**：找反复出现但**不在 tracker 里**的名词/术语
4. 每个高频新词反向回带到标的："这个变量如何影响标的？"

**关键**：不搜公司名，搜**社区里讨论的相关行业话题**。

**示例**：
- r/farming 反复出现 "anhydrous ammonia shortage" → 不在 CF tracker → 候选
- r/hardware 反复出现 "TSMC N2 yield issue" → 不在 QCOM tracker → 候选

### 机制 C（P1）：反向案例映射

**核心**：历史不会重复，但会押韵。

**方法**：
1. 找 5-10 个**同行业历史崩盘/暴涨案例**（同商业模式、同周期阶段）
2. 提取当时的驱动变量
3. 映射到当前标的：这个变量是否适用？前置条件是否具备？
4. 适用 → 候选变量（标注历史案例引用）

**示例**：
- 2008 化肥崩盘：天然气暴涨 + 飓风 + 需求萎缩 → 当前 CF 的前置条件是否在形成？
- 2018 半导体库存危机 → 当前 QCOM 是否有类似库存积压信号？
- 2022 密西西比河枯水 → 当前是否可能重演？

### 机制 D（P1）：研报反向扫描

**核心**：研报是双面工具——挖事实数据，忽略评级；找研报里**没有的东西**。

**方法**：
1. 收集 3-5 份标的及同行业研报
2. 扫描研报中的**事实性名词**（行业规模、产业链节点、技术参数、客户名单）——这些是分析师做了功课的成果
3. 词频对比：研报反复出现但 tracker 里没有的事实性名词 → 候选变量
4. **反向信号**：如果研报评级极端一致（全部"买入"），标记为反向预警
5. **覆盖度信号**：如果某标的研究覆盖度低（研报少）= 冷门机会信号

**禁止**：不挖评级、目标价、推荐（散户跟着做 = 接盘）。

### 机制 E（P2）：兄弟差异扫描

**核心**：同板块兄弟标的有的变量，当前标的没有 → 候选缺失。

**方法**：
1. 找 2-3 个同板块兄弟标的及其 tracker
2. 列出兄弟的变量清单
3. 对比：兄弟有而当前标的没有的变量 → 候选
4. 问"这个变量为什么不适用？"——真不适用，还是被忽略？

**示例**：
- CF 的变量 vs MOS、NTR 的变量 → MOS 有磷肥出口政策变量，CF 是否受影响？

### 机制 F（P2）：透镜反推

**核心**：不是"用透镜看标的"，而是"透镜成立依赖什么变量"。

**方法**：
1. 从 248 张透镜中，选 5-10 张与标的/板块相关的
2. 对每张透镜，问："如果用这个透镜看标的，**它依赖什么变量**？"
3. 检查这些依赖变量在 tracker 里是否覆盖
4. 未覆盖 → 候选变量

**示例**：
- "周期钟摆"透镜 → 依赖"市场情绪极端值"变量 → tracker 里有吗？
- "第二层次思维"透镜 → 依赖"市场共识 vs 真实"的偏差量化 → tracker 里有吗？

**联动**：此机制可与 `/wiki-lens` 联动，复用透镜匹配逻辑。

## 停止规则（4 个，触发任意一个就停）

| 停止条件 | 含义 | 判定方法 |
|---------|------|---------|
| **传导断链** | 候选变量到标的的路径无法量化 | 不能粗估"X 变动 10% → 影响 EPS $Y" |
| **信号孤立** | 只在 1 个来源出现，且 2 周内无第二次 | ≥2 个独立来源，或 1 个权威源 + 1 个社区源 |
| **已被覆盖** | 变量已在 registry 或 tracker 里 | 即使名字不同，本质是同一变量 |
| **跳数超限** | 超过软警告/硬上限 | 软警告 5 跳，硬上限 10 跳 |

### 跳数限制（详细）

| 跳数 | 价值 | 处理 |
|------|------|------|
| 1 跳 | 直接关联（已知为主） | 必经，快速通过 |
| 2-3 跳 | 主战场（80% 价值在这） | 重点投入 |
| 4-5 跳 | 精选远端 | 偶尔有金矿 |
| **5 跳（软警告）** | 进入低价值区 | 每多 1 跳必须显式回答"为什么这一跳比上一跳更可能找到盲区？"——答不出就停 |
| 6-10 跳 | 极端长链 | 罕见但存在 |
| **10 跳（硬上限）** | 强制终止 | 但记录 pending 线索，最后问用户 |

**重要区分**：跳数 ≠ 搜索次数。一次"跳"= 沿产业链走一个节点。一次跳可能同时产生多个候选（天然气节点 → 管道运力/储量/出口政策/LNG 设施等多个候选并行）。所以 10 跳限制不影响发现广度，只防止沿单条链条无限深入。

### 超出跳数 = 记录 pending，不静默抛弃

当达到硬上限 10 跳时：
1. **不要静默抛弃**未探索的线索
2. 将未探索的线索记录到 `pending_clues` 列表，每条包含：
   - 线索描述
   - 当前跳数
   - 为什么这条线索可能值得继续（一句话）
   - 预估继续挖掘需要的跳数
3. 所有工作完成后（Phase 7），**显式问用户**："以下是超出跳数限制的线索，是否继续挖掘？"
4. 用户决定继续 → 单独走该线索；用户说停 → 归档

## 时效性判断（按变量生命周期）

**核心规则**：不用绝对时间窗口一刀切。判断"信号采集时的状态"vs"当前状态"是否一致。

每个候选变量必须带 `state_snapshot`（采集时的状态值）。复核时只问一句：**"这个状态还成立吗？"**

| 生命周期 | 时效判断 | 例子 |
|---------|---------|------|
| **结构性**（3 年+） | 季度复盘"是否还成立" | 中美关系——3 年前的信号如果状态没变就不过时 |
| **周期性**（半年-2 年） | 跟踪周期位置 | 尿素价格——3 个月前的 $444 如果现在还是 $444 就不过时 |
| **季节性**（每年固定时段） | 提前 1-2 月激活 | 飓风季、农业种植季 |
| **事件性**（一次性） | 触发后归档（变 archive） | SpaceX IPO、Warsh 接任 |
| **脉冲性**（几小时-几天） | 基本不进变量层 | 马斯克推特——进事件日志，不进 registry |

**判断标准**：每个候选变量必须带 `state_snapshot`（采集时的状态值）。复核时只问一句："这个状态还成立吗？"——成立就保留，不成立就更新或归档。

## 候选池机制（两阶段）

```
挖掘产出
  ↓
候选池（tracker 本地的"待验证变量"区）
  ↓ 升级门槛触发
registry（全局可复用）—— 分配全局 ID + 写入板块模板
```

### 候选池位置

放在每个标的的 tracker 里（已有"待验证变量"机制，复用这个结构）。

### 升级到 registry 的门槛（同时满足）

1. **信号强度**：≥2 个独立来源出现（或 1 个权威源 + 1 个社区源）
2. **传导可量化**：哪怕只是数量级估算（"X 变动 10% → 影响 EPS $Y"）
3. **不重复**：与现有 registry 变量不重复（或是对现有变量的细化，需标注关系）
4. **持续观察**：≥2 周（事件性变量除外，可以立即升级）

### 升级时执行

1. 在 `registry.md` 分配新 ID（按板块规则）
2. 在对应 `sectors/{板块}.md` 模板添加（如果适用）
3. 在 `edges.md` 添加传导关系
4. 在 tracker 的"全局变量引用"表添加新 ID
5. 如果变量反复出现且影响多个标的 → 创建 `variables/{slug}.md` 深度档案

## 执行流程

### Phase 0：锚定（5 分钟）

1. 读取 registry、edges、板块模板、标 tracker
2. 列出**已有变量清单**（避免重复挖掘）
3. 确认标的所属板块
4. 准备 6 种机制的执行计划

### Phase 1：产业链漫游（机制 A，30-40 分钟）

主战场。按 BFS 走 1-5 跳（5-10 跳仅在长链标的时考虑）。

每个候选变量记录：
```
候选：天然气管道运力
来源机制：A 产业链漫游
当前跳数：2
传导路径：管道检修 → Henry Hub 局部涨价 → CF 成本↑
量化估算：CF 年用气量 X × 价格变动 Y = 成本影响 $Z
信号强度：待 Phase 5 汇总
state_snapshot：截至 2026-07-05，管道运力正常
生命周期：周期性
```

### Phase 2：社区异常词频扫描（机制 B，20-30 分钟）

抓取相关社区最近 30-90 天热帖，做词频分析。

每个候选变量记录：
```
候选：anhydrous ammonia shortage
来源机制：B 社区扫描
原始链接：r/farming/xxx（高赞帖，3 周前）
出现次数：在 30 天内 12 个独立帖子出现
传导路径：液氨短缺 → CF 部分产品供给紧张 → 价格支撑
state_snapshot：截至 2026-07-05，社区讨论活跃
```

### Phase 3：反向案例映射（机制 C，15 分钟）

找 5-10 个历史案例，提取变量，映射当前。

### Phase 4：研报反向扫描（机制 D，15 分钟）

收集研报，词频对比，找盲区 + 反向信号。

### Phase 5：兄弟差异扫描（机制 E，10 分钟）

对比同板块兄弟标的的变量清单。

### Phase 6：透镜反推（机制 F，10 分钟）

选 5-10 张透镜，反推依赖变量。

### Phase 7：信号汇总与合并

1. 汇总所有候选变量
2. **去重**：不同机制可能挖到同一变量（合并信号强度）
3. 按"信号强度"排序：
   - 出现源数量（≥3 个机制都挖到 = 极强信号）
   - 传导可量化程度
   - 时效性（state_snapshot 是否仍成立）
4. 对每个候选变量判断去向：
   - **立即升级**（事件性，信号强）→ 写入 registry
   - **进入候选池**（需观察 ≥2 周）→ 写入 tracker 待验证区
   - **丢弃**（信号孤立/传导断链/已被覆盖）

### Phase 8：超跳数线索用户决策（强制）

如果 Phase 1-6 中有线索达到硬上限 10 跳被终止：
1. 列出所有 pending 线索
2. **显式问用户**："以下是超出跳数限制的线索，是否继续挖掘？"
   - 列出每条线索：描述 / 当前跳数 / 为什么可能值得继续 / 预估继续跳数
3. 用户决定继续 → 单独走该线索
4. 用户说停 → 归档，不静默抛弃

### Phase 9：输出

1. **候选池更新**：在 tracker 的"待验证变量"区添加新候选（标注来源机制、信号强度、传导路径、state_snapshot）
2. **registry 升级**（如有立即升级的变量）：写入 registry + 板块模板 + edges
3. **tracker 全局引用更新**：在 tracker 的"全局变量引用"表添加新 ID
4. **挖掘报告**：在 tracker 的跟踪记录追加一条：
   ```
   - **YYYY-MM-DD**: wiki-mine 挖掘——发现 N 个候选变量（M 个进入候选池，K 个立即升级，P 个超跳数待用户决策）。来源：机制 A/B/C/D/E/F。最强信号：[变量名]（来源 A+B+D 三个机制同时挖到）。
   ```
5. **pending_clues 归档**：用户决策后，无论继续还是停止，都记录决策结果

## 反例黑名单

1. **不要用公司名搜新闻** —— 那是 top-down 搜索，不是 bottom-up 挖掘
2. **不要把分析师评级当信号** —— 评级是滞后指标，散户跟着做 = 接盘
3. **不要产出"全面分析"** —— 只产出**新发现的盲区变量**，已知变量不重复
4. **不要静默抛弃超跳数线索** —— 必须 pending + 问用户
5. **不要用绝对时间过滤信号** —— 按变量生命周期类型 + state_snapshot 判断时效
6. **不要跳过 Phase 8（用户决策）** —— 即使你认为线索不重要，也要列出来让用户决定
7. **不要在单机制就停止** —— 至少跑完机制 A+B（P0 必须），其他视情况
8. **不要混淆"搜索"和"挖掘"** —— 搜索是已知-验证，挖掘是未知-发现。如果开始时你知道要找什么，那就不是挖掘

## 时间预算

每个标的**不超过 1 个完整 session**：
- Phase 0 锚定：5 分钟
- Phase 1 产业链漫游：30-40 分钟（重点）
- Phase 2 社区扫描：20-30 分钟（重点）
- Phase 3 反向案例：15 分钟
- Phase 4 研报反向：15 分钟
- Phase 5 兄弟差异：10 分钟
- Phase 6 透镜反推：10 分钟
- Phase 7 信号汇总：10 分钟
- Phase 8 用户决策：5 分钟
- Phase 9 输出：10 分钟
