# wiki-lens — 透镜扫描

> **分层**：③ 挖掘类 · 主动找新角度
> **被调**：用户、wiki-digest（步骤 10.5 入库新文章后）、wiki-research（收尾步骤强制跑）、wiki-sweep（维度 L）
> **调用**：无（独立分析，产出跨界洞察。可选建议用户触发 wiki-research 深挖）

## 核心理念

AI 是生成式的，不会自发"创造"信息点之间的关联。蒸馏出的方法论卡片如果只躺在 sources/ 里不被调用，等于死知识。

本 skill 把每张蒸馏卡片的「作为透镜如何使用」字段作为**强制视角**，主动应用到任意事件/变量上，让方法论之间互相碰撞，产出单看任何一篇文章都得不出的新洞察。这是博主大脑的工作方式机制化。

**方法论是透镜，数据是焦点。没有焦点的透镜只是空转。** 每次透镜分析必须先搜集最新数据，用数据验证每个透镜的判断，不允许脱离数据做纯方法论推理。

**先透镜后skill，一次产出两层结论。** 透镜广扫产出跨界洞察（广），再自动衔接最相关的 skill 做深度分析（深），用户不需要二次手动调用。

## 触发条件

- 用户显式调用："用透镜看 XX 事件"、"lens 分析下"、"用方法论分析下 XX"
- wiki-digest 入库新文章后自动触发（作为步骤 10.5）
- wiki-research 完成研究后强制跑一次（作为收尾步骤）
- wiki-sweep 定期扫描时，对近期热门事件跑一次

## 前置检查

1. 环境变量 `WIKI_ROOT` 必须设置
2. `$WIKI_ROOT/wiki/analysis/*/skills/` 或 `~/.claude/skills/books/` 下必须有蒸馏书籍产出的 SKILL.md（含 `## L — 透镜` 字段），否则提示用户先蒸馏书
3. `$WIKI_ROOT/wiki/connections/` 目录存在

## 反例黑名单

1. **不要做标签重叠关联** — "A 也讲风险，B 也讲风险" 是废话关联，禁止产出
2. **不要用通用透镜套所有事件** — 必须基于事件特征匹配透镜的「适用信号」字段
3. **不要产出"全面分析"** — 只产出非显然关联，能不说就不说
4. **不要写没有 C 的 A+B 关联** — 透镜碰撞的产出必须有 C（新洞察），A+B 没出新东西就丢弃
5. **不要透镜数量超过 5 个** — 3-5 个最佳，太多稀释思考
6. **不要脱离数据做纯方法论推理** — 每个透镜的判断必须用真实数据支撑，不允许"大约"、"应该"等模糊表述

## 执行步骤

### 1. 解析输入

识别要分析的对象：
- **事件型**："拼多多 de minimis 取消"、"NVIDIA 自研 CPU"
- **变量型**："美债 10Y 利率突破 4.5%"
- **文章型**：刚入库的 sources 卡片路径

提取事件的关键特征（用于透镜匹配和数据搜集）：
- 事件类型（突破/崩盘/政策/财报/合作/冲突...）
- 涉及实体（公司/国家/资产/人物）
- 叙事方向（利好/利空/中性/反转）
- 极端性（是否处于极端位置）

### 2. 扫描透镜库

透镜来自两个来源，都需扫描：

**来源1（主要）：蒸馏书籍产出**
- 路径：`$WIKI_ROOT/wiki/analysis/*/skills/*/SKILL.md`
- 备份路径：`~/.claude/skills/books/*.md`
- 字段标记：`## L — 透镜` 或 `## L — 透镜 (Lens)`
- 透镜内容在该字段下，包含「适用信号」「强制问题」「关联触发」

**来源2（补充）：sources 卡片自带透镜**
- 路径：`$WIKI_ROOT/wiki/sources/`
- 字段标记：`## 作为透镜如何使用`

扫描操作：

1. 用 Grep 搜索两个路径的字段标记：`## L — 透镜` 和 `## 作为透镜如何使用`
2. 对每张匹配的卡片，提取其「适用信号」字段
3. 把事件特征与每张卡片的适用信号做匹配
4. 按匹配度排序，**选出 top 3-5 个透镜**

匹配标准（从严到松）：
- 直接命中：事件关键词出现在适用信号里
- 场景匹配：事件类型对应适用信号描述的场景
- 极端性匹配：事件处于极端位置时，自动匹配所有"极端/周期/逆向"类透镜

**至少包含 1 个跨领域透镜**（如果事件是投资的，至少有 1 个透镜来自兵法/心理学/系统思维等非投资领域）。强制跨界思考。

### 3. 数据搜集（强制）

> ⚠️ 本步骤不可跳过。没有数据的透镜分析不如不做。

在选定透镜后、逐透镜应用前，**必须搜索事件相关的核心数据**。数据是透镜的焦点——没有焦点，透镜只是空转。

#### 3.1 确定数据需求

根据事件类型和透镜特征，列出需要搜集的数据维度：

| 事件类型 | 必须搜集的数据维度 |
|---------|-----------------|
| 股票/资产 | 市值、PE/Forward PE/PEG、营收增速、市场份额、竞品数据 |
| 宏观经济 | 利率、通胀、就业、PMI、相关历史对比数据 |
| 行业趋势 | 市场规模、增速、渗透率、主要玩家份额、capex |
| 公司事件 | 最新财报数据、指引、管理层发言、竞争对手反应 |
| 政策/监管 | 政策细节、影响范围、时间表、相关历史案例 |

#### 3.2 执行搜索

使用 MCP 工具和 web search 搜集数据，遵循 CLAUDE.md 的「搜索时效性规则」：

1. **精确数据优先**：先用 MCP 工具（alphavantage/yahoo-finance/等）获取精确数值
2. **补充搜索**：用 web search 获取行业趋势、市场份额、竞品数据
3. **中英文双语**：同一主题至少中英文各搜一次
4. **搜索词含当前年份**：确保数据时效性

#### 3.3 建立数据表

把搜集到的数据整理成结构化表格，标注截至日期和可信度：

```markdown
## 关键数据（截至 YYYY-MM-DD）

| 指标 | 数值 | 截至 | 来源/可信度 |
|------|------|------|------------|
| 指标A | xxx | YYYY-MM-DD | 来源（高/中/低） |
| 指标B | xxx | YYYY-MM-DD | 来源（高/中/低） |
```

**数据质量要求**：
- 每个数据点必须标注**截至日期**
- 必须标注**可信度**（高/中/低，参见 CLAUDE.md 信息来源可信度评估）
- 优先使用官方文件、交易所数据等高可信度来源
- 低可信度数据不能单独支撑透镜判断，必须交叉验证

### 4. 逐透镜应用（结合数据）

对每个选中的透镜，按其「强制问题」字段逐条回答。**每条回答必须摆出证据**：哪个数据点支撑了你的判断，来源是什么，支撑方向是正向还是反向。产出结构：

```markdown
### 透镜 N：《书名-卡片名》

**判断**：<一句话结论，必须有观点，不能模棱两可>

**证据**：
- <数据点1> xxx（来源，截至 YYYY-MM-DD，可信度）→ 支撑判断
- <数据点2> xxx（来源，截至 YYYY-MM-DD，可信度）→ 支撑判断
- <数据点3> xxx（来源，截至 YYYY-MM-DD，可信度）→ ⚠️ 部分反驳 / 需注意
```

每个判断必须：
- 有明确观点（涨/跌/持续/反转/无关/重要/可忽略）
- **摆出至少 2 个证据**（数据点 + 来源 + 截至日期 + 可信度），标注每个证据是支撑还是反驳
- 不能是"这件事很重要需要关注"这种废话
- 不能用"大约"、"应该"等模糊表述替代精确数据
- 证据标注"⚠️ 部分反驳"的数据点不能忽略，必须展示并解释为什么不推翻结论
- 如果证据与判断矛盾，**以数据为准**，修改判断或标注偏差原因
- 每个证据的可信度至少有一个"高"（官方文件/交易所数据），全"低"可信度的判断不合格

### 5. 透镜碰撞

把所有透镜的判断并排放，找：
- **收敛点**：多个透镜指向同一结论 → 用数据强化该结论
- **分歧点**：透镜之间结论冲突 → 用数据判断哪个更接近现实
- **互补点**：不同透镜看到不同维度 → 拼成完整图景

碰撞产出格式：

```markdown
### 透镜碰撞

**收敛**：N 个透镜共同指向 → <结论>
- 证据：数据点1 + 数据点2 + 数据点3 均指向同一方向

**分歧**：透镜 A 说 X，透镜 B 说 Y
- 证据对比：透镜A依据 [数据点X]，透镜B依据 [数据点Y]
- 数据裁决：[数据点Z] 更支持 <哪个方向>，差异源于 <假设差异>

**新洞察（A+B+C+...→Z）**：<这是 wiki-lens 的核心产出>
- 单看任何一张卡片得不出的判断
- 必须有反直觉成分（让人"哦，原来如此"）
- 关联到具体决策（持仓/操作/关注点）
- **核心洞察的证据链**：摆出支撑洞察的核心证据（至少 2 个高可信度数据点），如果数据不支持，洞察不成立
```

### 6. 透镜→skill 自动衔接（先广后深）

透镜碰撞产出跨界洞察后，**自动选出 1-2 个判断最犀利/最需要行动的透镜，调用对应 skill 的 E 段（可执行步骤）做深度分析**，不需要用户二次手动调用。

#### 6.1 筛选深入对象

从步骤 4-5 的透镜中，按以下优先级选出 1-2 个做深入：

| 优先级 | 信号 | 说明 |
|--------|------|------|
| ★★★ | 透镜判断指向**明确行动**（该买/该卖/该止损/该启动） | 行动型判断最需要 skill 的步骤拆解 |
| ★★★ | 该透镜是碰撞**收敛点**（多个透镜指向它） | 收敛方向值得深挖 |
| ★★☆ | 透镜判断有**反直觉成分**（与市场共识相反） | 反直觉判断需要严谨论证 |
| ★★☆ | 该透镜的 skill 的 E 段**步骤清晰可执行** | 有些 skill 偏认知，E 段弱，优先选 E 段强的 |
| ★☆☆ | 用户在对话中**已表达关注**该维度 | 顺着用户兴趣深挖 |

**不要选超过 2 个**——深度分析耗时，2 个已经足够。

#### 6.2 读取 skill 的 E 段

对每个选中的透镜，找到其对应的完整 skill（透镜是 skill 的 L 段，skill 在 `~/.claude/skills/books/{书名}-{skill名}.md` 或 `$WIKI_ROOT/wiki/analysis/{书名}/skills/{skill名}/SKILL.md`）。

读取该 skill 的：
- **I 段**（方法论骨架）：深度分析的框架
- **E 段**（可执行步骤）：逐步执行
- **B 段**（边界）：检查是否适用

#### 6.3 执行深度分析

**用步骤 3 搜集的数据 + 透镜碰撞的洞察作为输入**，按 skill 的 E 段逐步执行。每步产出必须有数据支撑，不允许纯方法论推理。

产出格式：

```markdown
### 深度分析 N：《书名-skill名》

**为什么深入这个 skill**：<一句话说明为什么从透镜广扫中选择这个做深挖>

**E 段逐步执行**：

**步骤1：xxx**
- 执行结果：<结合数据的具体判断>
- 数据支撑：指标A = xxx（来源，截至 YYYY-MM-DD）

**步骤2：xxx**
- 执行结果：<结合数据的具体判断>
- 数据支撑：指标B = xxx（来源，截至 YYYY-MM-DD）

...

**B 段边界检查**：
- <边界条件1>：是否触发？<是/否，理由>
- <边界条件2>：是否触发？<是/否，理由>

**深度结论**：<一句话可行动结论，比透镜判断更具体更可操作>
```

#### 6.4 深浅结论交叉验证

把深度分析结论与步骤 5 的透镜碰撞洞察并排，检查：
- **一致**：深度分析强化了透镜洞察 → 提升信心，可行动
- **部分分歧**：深度分析修正了透镜洞察 → 以深度为准，记录差异原因
- **矛盾**：深度分析推翻了透镜洞察 → **必须展示矛盾**，以深度为准，透镜洞察降级为"初步判断"

## 质量红线

1. **每个透镜的判断必须有明确观点** — 模棱两可不合格，重做
2. **每个透镜的判断必须摆出证据链** — 至少 2 个证据（数据点+来源+截至日期+可信度），标注支撑/反驳方向，全"低"可信度证据不合格
3. **判断不能隐藏反面证据** — 遇到与结论矛盾的数据必须展示并解释，不能假装没看到
4. **新洞察必须有反直觉成分** — "拼多多关税取消对拼多多不利"是废话洞察，不合格
5. **新洞察必须通过数据验证** — 摆出支撑洞察的核心证据（至少 2 个高可信度），如果数据不支持，洞察不成立，不能写入 connections
6. **至少 1 个透镜来自事件所属领域之外** — 强制跨界
7. **关联卡片必须包含数据表** — 没有「## 关键数据」段的卡片不合格
8. **关联卡片必须标 type 和 lenses** — 否则无法被后续 sweep 检索
9. **碰撞产出不能只有 A+B 没 C** — 没新东西就别写 connections 卡片
10. **数据必须标注截至日期和可信度** — 不标注的数据点视为无效
11. **skill 深度分析不超过 2 个** — 选最值得深挖的，多了稀释深度
12. **skill 深度分析必须走完 E 段步骤** — 不能只读个结论就贴上来，必须逐步执行结合数据
13. **深浅结论矛盾时必须展示矛盾** — 不允许隐藏深度分析推翻透镜洞察的情况，以深度为准并记录差异

## 数据刷新规则

connections 卡片中的数据不是一成不变的。当 connections 卡片被引用或回顾时：

1. **检查数据时效**：如果「关键数据」表中的数据超过 CLAUDE.md 定义的时效窗口，标记为 `[⚠️ 数据过期]`
2. **刷新触发**：
   - 用户显式要求刷新
   - wiki-sweep 扫描发现数据过期
   - wiki-lens 再次分析同一事件时
3. **刷新操作**：搜索最新数据，更新数据表，标注 `updated: YYYY-MM-DD`
4. **判断是否需要更新结论**：如果数据变化显著（>20% 或方向性变化），重新审视透镜判断和新洞察

## 自动触发集成

### 集成到 wiki-digest（步骤 10.5）

- 扫描未应用透镜的 sources 卡片，建议跑一次透镜扫描
