# Review — 决策复盘流程

## 触发条件

- 用户说"复盘"、"回顾决策"、"review"
- 系统定时任务触发（cron）

## 前置检查

1. 检查当前工作目录是否符合 wiki 规范：需存在 `wiki/` 目录
2. 检查 `decisions/` 目录是否存在且包含决策记录
3. 如果不满足，报错并提示用户

## 执行步骤

### 1. 扫描待复盘决策

读取 `decisions/` 下所有文件，筛选：
- `status: open`
- `review_date <= 今天`

### 2. 判断是否有待复盘项

**没有待复盘项时**：
- 回复"当前没有到期的决策需要复盘。"
- 列出最近的几条决策及其 review_date，供参考
- 停止执行

**有待复盘项时**：
- 统计数量，告知用户："有 N 条决策到期待复盘。"
- 逐条进入复盘

### 3. 逐条复盘

对每条待复盘决策：

1. **展示决策背景**：
   - 决策内容、当时理由、预期
   - 关联的研究报告（如有 context 字段）

2. **请用户回顾**：
   - 实际发生了什么？
   - 和预期一致吗？
   - 有什么教训？

3. **记录复盘内容**：
   - 将用户反馈写入该决策文件的 `## 复盘` 部分
   - 更新 `status: reviewed`
   - 更新 frontmatter 中的 `reviewed_date`

4. **追问**（可选）：
   - 这条决策是否需要继续跟进？如果是，建议新的 review_date
   - 是否有共性教训可以记录？

### 4. 复盘总结

所有到期决策复盘完成后：

- 汇总本次复盘了几条
- 提炼共性教训或模式（如有）
- 建议是否需要更新 `context/` 中的个人偏好（如投资风格、风险偏好等）
- 追加日志到 `wiki/log.md`：
  ```
  - YYYY-MM-DD: 完成决策复盘，共 N 条。共性发现：...
  ```

## 用户交互规则

- **逐条展示**：一次只展示一条决策，等用户回复后再继续下一条
- **简洁展示**：只展示决策、理由、预期三个部分，不展示完整文件
- **引导但不替用户总结**：可以提问引导，但复盘内容由用户说，AI 记录
- **允许跳过**：用户可以说"这条先跳过"，留待下次复盘

## 通知输出

复盘完成后，在回复末尾**必须**输出一行通知摘要（供 cron wrapper 解析）：

```
NOTIFY: review 完成：复盘 N 条决策
```

- 如果有待复盘项：`NOTIFY: review 完成：复盘 2 条决策，待处理 1 条`
- 如果没有到期项：`NOTIFY: review 完成：当前无到期决策`
- 复盘是交互式流程，cron 触发时如果需要用户输入，只记录日志并通知用户有待复盘项，不在 cron 中强制交互

## 错误处理

- **decisions/ 为空**：告知用户还没有决策记录
- **决策文件格式异常**：跳过该文件，告知用户
- **用户中途退出**：已复盘的部分正常保存，未复盘的保持 open 状态

## 路径约定

所有路径基于当前工作目录（相对路径）。当前目录必须有 `wiki/` 和 `decisions/` 目录才算有效。
