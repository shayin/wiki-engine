# Review — 决策复盘流程

## 触发条件

- 用户说"复盘"、"回顾决策"、"review"
- cron wrapper 传入粗筛结果时自动触发

## 前置检查

1. 检查 `wiki/` 目录是否存在
2. 检查 `decisions/` 目录是否存在且包含 .md 文件
3. 不满足时报错："当前目录不是有效的 wiki 根目录，或还没有决策记录。"

## 执行模式

### 模式 A：交互模式（用户手动触发）

完整扫描 + 逐条交互复盘。

### 模式 B：精析模式（cron wrapper 传入粗筛结果）

当 prompt 中包含「粗筛已发现以下问题」时激活：
- 只读取粗筛命中的决策文件
- 不进行交互式复盘（无用户输入）
- 列出到期项并写入 log.md，通知用户手动触发

### 粗筛结果格式

由 review-check.sh 生成，每行一条：
```
decisions/xxx.md|决策标题|2026-05-01
```
字段：文件路径 | 标题 | review_date

## 执行步骤

### 1. 扫描待复盘决策

**交互模式**：读取 `decisions/` 下所有文件，通过 frontmatter 筛选 `status: open` 且 `review_date <= 今天`。如果 decisions/ 下没有 .md 文件或所有决策都不是 open 状态，跳到步骤 2 的"无待复盘"分支。

**精析模式**：只读取粗筛结果中的文件路径。文件不存在则跳过该条。

### 2. 判断是否有待复盘项

**无待复盘项**：
- 回复"当前没有到期的决策需要复盘。"
- 列出所有 open 状态决策的标题和 review_date（如有），供参考
- 输出 `NOTIFY: review 完成：当前无到期决策`
- 停止执行

**有待复盘项（交互模式）**：
- 告知用户："有 N 条决策到期待复盘。"
- 进入步骤 3 逐条复盘

**有待复盘项（精析模式）**：
- 列出每条的标题、review_date、距今天数
- 写入 `wiki/log.md`：`- YYYY-MM-DD HH:mm: [auto] N 条决策到期待复盘（{标题列表}），等待手动触发 /wiki-review`
- 输出 `NOTIFY: review 完成：N 条决策到期待复盘，请手动触发复盘`

### 3. 逐条复盘（仅交互模式）

对每条待复盘决策，一次只展示一条：

1. **展示背景**（决策、理由、预期三部分，不展示完整文件）：
   ```
   📋 决策：{决策内容}
   📝 理由：{当时理由}
   🎯 预期：{预期结果}
   📅 决策时间：{date}  到期：{review_date}
   ```

2. **请用户回顾**（引导性问题）：
   - "实际发生了什么？"
   - "和预期一致吗？"
   - "有什么教训？"
   - 用户可以说"跳过"留待下次

3. **记录复盘**：
   - 在决策文件的 `## 复盘` 部分写入用户反馈
   - 更新 frontmatter：`status: reviewed`，追加 `reviewed_date: YYYY-MM-DD`

4. **追问**（可选）：
   - 是否需要继续跟进？如需要，建议新的 review_date
   - 复盘中是否有共性模式？

### 4. 复盘总结（仅交互模式）

全部复盘完成后：
- 汇总本次复盘条数
- 提炼共性教训（如有）
- 如发现涉及 `context/` 的偏好变化（如投资风格、风险偏好），建议用户更新
- 写入 `wiki/log.md`：`- YYYY-MM-DD HH:mm: 完成决策复盘，共 N 条。共性发现：...`
- 输出 `NOTIFY: review 完成：复盘 N 条决策`

## 用户交互规则

- 一次一条，等用户回复后再继续
- 引导但不替用户总结——复盘内容由用户说，AI 记录
- 允许"跳过"

## 错误处理

- **decisions/ 为空**：告知"还没有决策记录"
- **决策文件 frontmatter 缺少 status/review_date**：跳过该文件，告知用户
- **用户中途退出**：已复盘的正常保存，未复盘的保持 open

## 路径约定

所有路径基于当前工作目录（相对路径）。不使用配置文件，不向上查找。
