# AI Wiki Engine

基于 Claude Code 的个人知识管理系统。核心理念：**用户只做四件事——存、问、研、学**，中间的整理、分析、跟踪、关联发现全交给 AI。

## 核心路径

### 日常（用户触发）

| 路径 | 输入 | AI 做什么 |
|------|------|----------|
| **存** | 发一个链接 | 抓取全文 → 生成知识卡片 → 归档 → 更新索引，同类文章多了自动聚合主题 |
| **问** | 直接提问 | 两轮搜索定位 → 精读判断相关性 → 基于已有资料回答（附来源），不知道就说不知道 |
| **研** | 说"研究XX"/"研XX" | 开题 → 搜集资料 → 分析 → 输出报告 → 入库 → 识别需要持续跟踪的发现 |
| **学** | 说"学习XX" | 扫描知识库 → 组织学习路径（基础→深入→关联→内化）→ 逐项标记进度 |
| **记** | 说"加个待办" | 自动分类（工作/个人）→ 管理生命周期（添加/完成/归档），跟踪项按课题分组 |

### 自动化（定时执行）

| 路径 | 频率 | 做什么 |
|------|------|--------|
| **入库** | 每天 23:00 | 检查 inbox/，有新文章就自动处理（唯一全自动执行的任务） |
| **扫描** | 每天 23:15 | 十一维度扫描知识库健康度（跟进项 + 知识库维护 + 数据时效 + 关联发现），发现问题后提醒用户确认 |
| **复盘** | 每天 23:15 | 扫描到期决策，提醒用户回顾当初判断对不对 |
| **提醒** | 每天 11:00 & 23:00 | 统计待办数和跟踪项，写入通知队列 |

## 快速开始

### 1. 部署

```bash
git clone https://github.com/shayin/wiki-engine.git
cd wiki-engine

# 一键安装（创建目录 + 安装 skills + 配置 crontab）
./install.sh ~/my-wiki --with-cron

# 或分步安装
./install.sh ~/my-wiki           # 只安装目录和 skills
./install.sh ~/my-wiki --with-cron  # 后续再加 crontab
```

部署后的目录结构：

```
my-wiki/
├── inbox/              # 待处理文章（发给 AI 链接后暂存于此，digest 处理后清空）
├── raw/                # 原文归档（按月 YYYY-MM/，不可变，处理后从 inbox 移入）
├── context/            # 个人上下文（用户自行维护，AI 研究时只读引用）
│   └── finance.md      #   如：投资风险偏好、持仓、关注领域
├── decisions/          # 决策日志（研究产出或用户口述后 AI 写入，定期复盘）
│   └── YYYY-MM-DD-标题.md  #  每条决策：决定/理由/预期/复盘
├── todos/              # 待办管理（AI 管理，支持 remind 定时提醒）
│   ├── active.md       #   活跃待办 + 闹钟 + 跟踪项（按课题分组）
│   └── archive/        #   按周归档（YYYY-WXX.md，含已完成和顺延项）
├── .cron/              # 定时任务系统（每分钟检查，合盖断网后自动补跑）
│   ├── config.sh       #   用户配置（推送密钥 + 调度时间，不入 Git）
│   ├── pending.md      #   通知队列（定时任务写入，AI 对话时提醒用户）
│   ├── sweep-issues.md #   sweep 扫描问题持久化清单（未解决则持续提醒）
│   ├── logs/           #   执行日志（按日 YYYY-MM-DD.log）
│   │   ├── last-runs.txt   #  各任务最后执行时间戳（用于补跑判断）
│   │   └── reminded.txt    #  已推送提醒记录（防重复推送）
│   └── scripts/        #   定时任务脚本（由 cron-check.sh 统一调度）
│       ├── cron-check.sh       # 入口：每分钟运行，按时间触发子任务
│       ├── wiki-cron.sh        # AI 任务 wrapper（调 Claude 执行 skill）
│       ├── bark-push.sh        # 推送工具（Bark + 微信，异步发送）
│       ├── todo-remind.sh      # remind 匹配 + 闹钟检查 + 早晚汇总
│       ├── inbox-check.sh      # digest 粗筛（检查 inbox 是否有文件）
│       ├── sweep-check.sh      # sweep 粗筛（零 token 检查八维度）
│       └── review-check.sh     # review 粗筛（检查到期决策）
├── wiki/               # 知识库核心（AI 维护，用户以问代查）
│   ├── index.md        #   总索引（AI 入口点，每次操作先读此文件）
│   ├── log.md          #   处理日志（只追加，记录所有入库操作）
│   ├── sources/        #   知识卡片（每篇文章一张，含摘要/要点/标签/关联）
│   ├── topics/         #   主题页面（同标签文章跨篇总结，有关联即创建）
│   ├── connections/    #   跨课题关联（不同专题间交叉分析，产出新洞察）
│   │   ├── README.md   #     目录说明和格式定义
│   │   └── index.md    #     关联索引（按类型分组：因果/共振/矛盾/对冲）
│   ├── learning/      #   学习计划（把知识库内容组织成有序学习路径，追踪进度）
│   └── analysis/       #   课题研究工作区（每课题一个子目录）
│       └── {课题名}/
│           ├── plan.md       # 研究计划（问题定义、已知/未知、搜索方向）
│           ├── materials/    # 搜集的原始资料
│           ├── notes.md      # 研究笔记（中间产出、信息缺口）
│           ├── report.md     # 研究报告（结论、来源、未解决问题）
│           └── follow-ups/   # 持续跟踪项（需定期更新的发现）
├── CLAUDE.md           # 治理文件（AI 行为规则，定义所有流程和格式）
└── wiki-engine/        # 引擎源码（Git 仓库，含 skills/模板/安装脚本）
    ├── install.sh      #   一键安装（创建目录 + 安装 skills + 配置 cron）
    ├── skills/         #   CC Skills 源码（安装到 ~/.claude/skills/）
    ├── template/       #   目录和文件模板（安装时复制到目标目录）
    ├── config/         #   配置文档
    └── scripts/        #   工具脚本
```

### 3. 配置定时任务

```bash
# 添加到 crontab（每分钟检查一次）
crontab -e
# * * * * * cd /path/to/my-wiki && .cron/scripts/cron-check.sh
```

### 4. 配置调度时间（可选）

编辑 `.cron/config.sh`，所有调度时间均可自定义：

```bash
# digest: 每天处理 inbox 的时间
DIGEST_TIME="23:00"

# sweep: 知识库健康度扫描（星期 + 时间）
SWEEP_DAY="Sat"
SWEEP_TIME="23:15"

# review: 决策复盘（每天扫描到期决策）
REVIEW_TIME="23:15"

# todo-remind: 待办提醒（多个时间逗号分隔）
TODO_TIMES="11:00,23:00"

# Bark 推送（留空则不推送手机通知）
BARK_KEY=""
BARK_GROUP="Wiki"
BARK_SERVER="https://api.day.app"

# 微信推送（留空则不推送到微信）
WECHAT_ID=""
WECHAT_PUSH_KEY=""
WECHAT_PUSH_SERVER=""
```

## 自动化任务

所有定时任务由 `cron-check.sh` 统一调度（每分钟检查，按时间触发），支持合盖/断网后自动补跑。调度时间在 `.cron/config.sh` 中配置：

| 任务 | 默认调度时间 | 做什么 | token 消耗 |
|------|------------|--------|-----------|
| **digest** | 每天 23:00 | 检查 inbox/，有新文章就自动处理入库 | inbox 非空才消耗 |
| **sweep** | 每天 23:15 | 十一维度扫描知识库健康度（跟进项 + 维护 + 数据时效 + 关联发现） | 默认 AI 直调 |
| **review** | 每天 23:15 | 扫描到期决策，提醒复盘 | 有到期决策才消耗 |
| **todo-remind** | 每天 11:00 & 23:00 | 统计待办数写入 pending.md 提醒用户 | 零 token |
| **todo-remind** | 每分钟 | 检查带 `remind` 标记的待办，时间到了推送明细 | 零 token |

任务执行流程：

```
cron-check.sh（每分钟，零 token）
  ├── todo-remind 每分钟检查带 remind 标记的待办 → 有匹配则推送明细
  ├── 未到调度时间 → 跳过（静默）
  ├── 到时间 + 无事可做 → 跳过（零 token）
  └── 到时间 + 有工作 → 调用 wiki-cron.sh
      ├── 阶段1: shell 粗筛
      │   ├── ALL CLEAR → 记录时间戳，结束（零 token）
      │   └── 发现问题 → 阶段2
      └── 阶段2: Claude 精析 → 写 pending.md 通知用户
```

## Skills

| Skill | 触发方式 | 说明 |
|-------|---------|------|
| `wiki-ask` | 直接提问、"问一下" | 搜索知识库，基于已有资料回答问题 |
| `wiki-digest` | 发链接、说"收录"/"处理" | 抓取文章、生成知识卡片、归档原文、更新索引 |
| `wiki-research` | 说"研究"/"研"/"调研"/"了解" | 开题→搜集→分析→报告→入库→跟进 |
| `wiki-todo` | 说"加个待办"/"完成了"/"看看待办" | 添加、完成、删除、编辑、归档待办 |
| `wiki-review` | 说"复盘"/"回顾"，或 cron | 扫描到期决策，逐条复盘，记录教训 |
| `wiki-sweep` | 说"扫一下"/"检查遗漏"/"整理知识库"，或 cron | 十一维度扫描知识库健康度（跟进项 + 维护 + 数据时效 + 关联发现） |
| `wiki-update` | 说"更新XX课题"/"刷新XX" | 搜索最新数据，补充到报告和材料中，标注变更 |
| `wiki-learn` | 说"学习XX"/"过一遍XX"/"看看XX相关内容"/"继续学习"/"看到哪了" | 把知识库内容组织成有序阅读路径，追踪进度 |
| `wiki-push` | 说"推送到微信"/"发微信"/"微信通知" | 推送消息到微信，支持空内容追问和错误分类 |

## 核心流程详解

### 文章收录（Digest）

```
用户发链接 → 抓取全文 → inbox/ → 生成知识卡片 → wiki/sources/ → 原文归档 raw/ → 更新索引
```

- 每篇文章生成一张知识卡片：一句话摘要、关键要点、标签、关联文章
- 有实质关联的文章自动生成主题页面（topics/），无需等待固定数量
- 入库时检查跨课题关联，发现时生成关联分析卡片（connections/）
- 入库时自动检查是否与已有跟进项相关
- 支持批量 URL 和用户自写文章（`source: self`）

### 课题研究（Research）

```
用户说"研究 XX" → 开题确认 → 搜集资料 → 研究分析 → 输出报告 → 入库 → 生成跟进项
```

- 研究产出分两类：
  - **知识型**（方法论、原理）→ 入库到 sources/
  - **决策型**（个股分析、个人建议）→ 保留在 analysis/ + 创建决策记录
- 研究完成后**必须**向用户确认跟进项（数据变化、指标跟踪等）
- **数据时效性**：核心量化数据标注"截至 YYYY-MM-DD"，过时数据触发更新搜索（区分回溯素材 vs 当前状态）
- 自动读取 `context/` 下的个人上下文，给出针对性建议

### 待办管理（Todo）

```
说"加个待办" → 写入 todos/active.md → 工作/个人分区 + 长期项
说"完成了XX" → 移到已完成区 → 周归档时汇总
```

- 自动判断工作/个人分类
- 长期待办关联跟进项和跟踪项
- 跟踪项按课题分组（如"拼多多"、"高通"）
- 支持 `remind` 定时提醒（每分钟检查，时间到了推送明细）：
  - `remind:HH:MM` — 每天提醒
  - `remind:DAY HH:MM` — 每周指定日提醒
  - `remind:M/D HH:MM` — 指定日期提醒一次

### 闹钟（Alarm）

闹钟与待办分开管理（在 `## 闹钟` 区），不需要完成，到点推送就完事。

```
说"设个闹钟"、"提醒我XX" → 写入 ## 闹钟区 → 到点推送
说"有哪些闹钟" → 显示所有闹钟
```

- 根据语义自动判断一次性或循环（"每天/每周X" → 循环，其余 → 一次性）
- 一次性闹钟触发后自动删除
- 支持格式：
  - `alarm:HH:MM` — 每天响
  - `alarm:DAY HH:MM` — 每周指定日响
  - `alarm:M/D HH:MM` — 指定日期响一次（响后自动删除）

### 遗漏扫描（Sweep）

十一维度检查知识库健康度，分四部分：

**跟进项**（需要用户确认后创建）：

| 维度 | 检查内容 |
|------|---------|
| A. 研究缺跟进 | 报告有值得跟踪的发现但没有 follow-up |
| B. 文章缺主题 | 同标签 >=3 篇但没创建 topic 页面 |
| C. 决策缺跟踪 | 决策有量化预期但没有跟踪机制 |
| D. 待办不一致 | follow-up 和长期待办不对应 |

**知识库维护**（可一键修复）：

| 维度 | 检查内容 |
|------|---------|
| E. Topic 缺更新 | topic 页面未关联同标签的新文章 |
| F. 标签碎片 | 同一概念用了不同标签（如"PDD" vs "拼多多"） |
| G. 孤立文章 | 无标签、无关联、未被任何 topic 引用 |
| H. 断链 | `[[]]` 链接指向不存在的文件 |
| I. 待办错分 | 待办放在通用区但应归入跟踪项（如知识库相关待办放在工作/长期） |

**数据时效**：

| 维度 | 检查内容 |
|------|---------|
| J. 数据过时 | 研究材料中核心数据超过时效窗口（经济指标>月度、公司数据>季度） |

**关联发现**：

| 维度 | 检查内容 |
|------|---------|
| K. 跨课题关联 | 不同课题间存在未记录的关联（因果/共振/矛盾/对冲） |

### 决策复盘（Review）

```
定时扫描 → 找到到期决策 → 展示当时的决策/理由/预期 → 用户回顾实际结果 → 记录教训
```

### 知识问答（Q&A）

```
用户提问 → 提取关键词 → 搜索 sources/topics/analysis/decisions → 精读判断相关性 → 基于知识库回答（附来源）
```

- 两轮搜索：先定位候选，再精读判断
- 按需读取 `context/` 个人上下文，给出针对性建议
- 三种回答模式：有内容（附来源）、部分有（标注缺口）、无内容（建议研究）
- 搜索不到时沿 `related` 双链扩展查找

## 通知机制

定时任务在无人值守时运行，通过 **pending.md 通知邮箱** 将结果传递给用户，同时支持 **Bark 手机推送** 和 **微信推送**：

### 推送渠道

| 渠道 | 说明 | 配置项 |
|------|------|--------|
| **Bark** | iOS 推送通知 | `BARK_KEY` |
| **微信** | 企业微信消息推送 | `WECHAT_ID` + `WECHAT_PUSH_KEY` |

两个渠道独立配置，可以同时启用，也可以只用其中一个。推送内容一致，格式为"标题: 正文"。

### 通知流程

```
cron 任务执行 → 发现问题 → 写入 .cron/pending.md
                                    ↓
用户打开 CC 对话 → CLAUDE.md 注入规则 → 检查 pending.md
                                    ↓
有未读通知 → 回复末尾附上提醒（带分隔线）
用户说"知道了" → 清空 pending.md（标记已读）
```

| 写入者 | 场景 | 通知内容 |
|--------|------|---------|
| wiki-digest | inbox 有新文章自动处理完 | 处理了 N 篇文章 |
| wiki-sweep | 发现遗漏跟进项或知识库维护项 | 需要确认的问题列表（可一键修复维护项） |
| wiki-review | 发现到期决策 | 需要复盘的决策列表 |
| todo-remind | 每天 11:00 & 23:00 | 待办数统计 |

**关键设计**：定时任务只通知，不自动修改知识库。所有修复动作需要用户确认后才执行。

## 设计原则

1. **用户只做输入，AI 负责中间过程**：发链接、提问题、说研究方向，其余交给 AI
2. **以问代找**：知识库是给 AI 读的，用户只需提问，不需要自己翻文件
3. **知识三层递进**：单篇文章（sources）→ 专题内聚合（topics）→ 跨课题关联（connections），层级越高信息密度越大
4. **关联创造价值**：单篇内容搜索引擎都有，不同信息交叉分析产生的新洞察才是知识库的核心差异
5. **知识库和个人数据分离**：wiki/ 是关于世界的知识，context/ 是关于用户自身的信息，decisions/ 是决策
6. **标签自动管理**：AI 自动打标签，有实质关联的文章自动聚合成主题
7. **决策可复盘**：每条决策有 review_date，定期回顾实际结果，持续改进判断力
